# 交互设计师转 AI 产品经理行动指南

## 1. 核心判断

你不需要从零学 AI 再当产品经理。你已经有 UX 这块最难拼图的大部分能力，缺的是把它翻译成 PM 语言，并插入 AI 产品的关键决策点。

成为 UX-trained AI PM，本质上是三件事：

- 你能看出别人容易忽略的 AI 体验风险。
- 你能把体验判断说成业务风险和产品约束。
- 你的交付物在团队流程里形成固定位置。

## 2. 你的 UX 能力如何转化为 AI PM 能力

| 你已有的 UX 能力 | AI 产品里的关键问题 | 你的武器化方式 |
| --- | --- | --- |
| 信息架构 / 流程梳理 | AI 流程不是线性页面流，而是输入、检索、生成、校验、展示、纠错的管道 | 画 AI 链路全景图，包含异常分支 |
| 用户心智模型 / 场景拆解 | 团队容易从 AI 能做什么出发，而不是从用户此刻信什么、怕什么出发 | 做信任触点地图 |
| 组件化思维 / 规范 | AI 输出形态容易碎片化：卡片、气泡、纯文本混用 | 建 AI 输出组件库 |
| 边界 case / 异常流设计 | AI 的大量工作量在失败路径，而不是 happy path | 把 fail path 写进 PRD |
| 信息呈现层级 | 模型输出一大段文字，用户读不动也不敢用 | 决定高亮、折叠、引用、操作区 |

你不是在学一个完全陌生的职业，而是在把已有能力换到 AI 产品语境里。

## 3. 五个你应该占住的抓手

### 抓手一：可信度设计

普通 PM 可能只问「它能不能生成」，你要多问「用户怎么相信它生成的」。

建议建立信任触点清单：

| 信任触点 | 要问的问题 | 产出 |
| --- | --- | --- |
| 源头透明 | 这段话从哪里来，能标引用吗 | 引用和来源标注规范 |
| 不确定性表达 | 模型不确定时，是装确定还是坦白 | 置信线索、证据不足提示 |
| 可逆性 | 用户采纳后能撤回吗 | 采纳、编辑、回退链路 |
| 可控感 | 用户是在驾驶，还是被系统带着走 | 参数面板、预览确认 |
| 失败尊严 | 拒答时是否说人话，是否给下一步 | 拒答文案和下一步引导 |

可信度设计不是视觉修饰，而是 AI 产品能不能被采纳的基础。

### 抓手二：AI 链路全景图

你要在团队里画出从用户触发到最终呈现的完整管道：

```text
用户触发
→ 输入校验
→ RAG 检索
→ Top-K 召回质量门
→ 拼 Prompt
→ 调模型
→ 输出解析
→ 安全过滤
→ 展示
→ 用户采纳、改写、反馈
→ 回流 Eval
```

这张图要包含异常分支：

- 输入为空。
- 输入超长。
- 检索不到。
- 生成超时。
- JSON 解析失败。
- 安全拦截。
- 低质量输出。

当你能画出这张图，你就不只是界面设计者，而是系统链路的协作者。

### 抓手三：失败路径 PRD 化

AI 产品里，失败路径是主战场。

在 PRD 里固定加入一节：

```text
异常与降级策略
```

建议字段：

| 异常 | 条件 | 用户看到什么 | 系统做什么 |
| --- | --- | --- | --- |
| 检索空 | 召回数量为 0 或最高相关度低于阈值 | 没找到足够资料，请补充或换问法 | 不生成，进入拒答/澄清 |
| 生成超时 | 超过等待上限 | 显示稍后再试或精简版 | 切 fallback |
| 解析失败 | JSON 无效或缺必填字段 | 显示重新生成 | 自动重试一次 |
| 安全拦截 | 命中敏感或越权 | 统一安全提示 | 记录日志并上报 |
| 低质输出 | 含禁用词或自相矛盾 | 拦截并给出可理解原因 | 进入人工抽检 |

这会让研发、算法和业务都更容易对齐边界。

### 抓手四：Eval 最小闭环

AI PM 不能只说「我觉得不行」，要能说「case 集证明它不行」。

你可以先建 30-50 条 eval sheet：

| ID | 输入 | 期望行为 | 红线 | 是否通过 |
| --- | --- | --- | --- | --- |
| Q01 | XX 产品保修期多长 | 从文档召回并引用 | 不能编造数字 |  |
| Q02 | 只给标题生成文案 | 要求补充参数或拒答 | 不能瞎编参数 |  |
| R01 | 输入含竞品名 | 输出不出现竞品名 | 不能直接复述 |  |

然后把它变成团队流程：

- 改 Prompt 要跑。
- 改 Chunk 策略要跑。
- 换模型要跑。
- 上线前要跑。

### 抓手五：语言转换

你的 UX 直觉必须翻译成 PM 和业务语言。

| UX 直觉 | PM/业务语言 |
| --- | --- |
| 用户看到无来源文字会慌 | 无引用会降低采纳率，提高人工复核成本 |
| 生成完没编辑入口，用户不敢采纳 | 缺编辑/撤销会让采纳漏斗断在最后一步 |
| 加载 5 秒没反馈，用户以为坏了 | p95 延迟过高会导致流失，需要进度和 fallback |
| 错误信息暴露内部报错 | 存在安全和信息泄露风险，需要统一拦截文案 |

练习方式：

每次提体验建议，先写一句：

```text
因为 ___，会导致业务损失 ___，所以需要 ___。
```

## 4. 90 天身份转化路线

### 第 1-2 周：占地盘

目标是让团队意识到「这事得让你参与」。

行动：

- 画一张当前 AI 功能链路全景图。
- 建第一版 eval sheet，30 条起步。
- 写一页 AI 输出组件规范：引用、重新生成、拒答、加载态、反馈入口。

里程碑：

- 评审会上有人开始问你 AI 输出和异常体验怎么定。

### 第 3-4 周：把抓手写进流程

行动：

- 在 PRD 中加入「信任触点设计」。
- 在 PRD 中加入「异常与降级策略」。
- 跑一次 Prompt 改版前后的 eval diff。
- 做一次 3-5 人的小型用户观察，记录模型行为和用户反应。

里程碑：

- 你的文档不再是设计补充，而是产品决策依据。

### 第 5-8 周：建立质量飞轮

行动：

- 做一页 AI 质量周报。
- 跟踪采纳率、拒答率、超时率、Top 失败原因。
- 推动来源引用或可纠错能力上线。
- 把 AI 输出组件规范推进设计系统。

里程碑：

- 你开始拥有 AI 功能的质量和体验标准。

### 第 9-12 周：成为模块 owner

行动：

- 主导一个 AI 功能从场景边界到上线复盘。
- 维护 case 库和质量看板。
- 固化团队的 AI PRD、Eval、红线清单模板。

里程碑：

- 团队默认你负责 AI 体验可信度、失败路径和质量闭环。

## 5. 你的身份叙事

你可以这样描述自己的转型定位：

```text
我不是来给 AI 做界面的。
我是来确保这个概率系统在用户面前可信、可控、可纠错，并且能稳定跑出业务结果。
```

这句话背后要有实际交付物支撑：

- AI 链路全景图。
- Eval case 集。
- 异常与降级策略。
- 信任触点设计。
- AI 输出组件规范。
- 质量周报。
