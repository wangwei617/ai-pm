# Agent 产品结构与治理

## 0. 一句话抓住本质

Agent 不是有性格的 AI，而是一个受控循环：模型每一步决定要不要调用工具，Harness 决定它能碰什么、跑多远、什么时候刹车，Trace 记录它做过什么。

## 1. 一句话定义

Agent 是一个在循环里运行的 LLM。它被允许在特定边界内使用工具去感知、查数据或做事，并由控制器/外壳决定什么时候继续、什么时候停止、什么时候叫停。

普通 LLM 调用是：

```text
输入 → 一次生成 → 输出结束
```

Agent 是：

```text
输入 → 想下一步 → 调工具 → 拿结果 → 再想 → 再调工具 → 最终输出/行动
```

Agent 的本质不是「更聪明」，而是多了：

- 工具。
- 循环。
- 停止条件。
- 权限边界。
- 可追溯日志。

## 2. Agent 的产品类比

可以把 Agent 想成一个受控助理。

用户说：

```text
帮我查清订单 #123 的物流，然后给客户发一条安抚短信。
```

Agent 的过程是：

1. 理解任务：需要先查物流。
2. 调用工具：查询物流。
3. 把工具返回结果写入上下文。
4. 判断信息是否足够。
5. 判断是否允许发短信。
6. 如果需要用户确认，先展示预览。
7. 用户确认后，再调用发送工具。
8. 返回完成结果。

关键点：Agent 不是有自我意识的 AI。它只是一个在边界内运行的工具调用循环。

它能做什么，不应该靠模型「自觉」，而应该靠 Harness 规定。

## 2.1 用助理比喻讲明白

Agent 像一个坐在工位上的助理。

他有：

- 脑子：理解任务，决定下一步。
- 电话：调用外部工具。
- 登记册：记录当前状态。
- 岗位制度：规定哪些电话能打、哪些事要审批。
- 工作日志：记录每一步做了什么。

助理不是因为「人品好」才不乱发消息，而是因为公司制度不允许。

Agent 也一样：不能靠模型自觉，必须靠 Harness 约束。

## 3. Agent 的五个最小构件

### A. 大脑：LLM

LLM 负责理解当前状态，并决定下一步。

它通常决定：

- 是否继续。
- 是否调用工具。
- 调哪个工具。
- 工具参数是什么。
- 是否直接回复用户。

典型结构：

```json
{
  "thought": "需要先查订单状态",
  "tool": "query_order",
  "tool_input": {
    "order_id": "123"
  }
}
```

产品上要注意：LLM 可以建议下一步，但不能绕过权限、确认和工具白名单。

### B. 手：Tools / Skills

Tools / Skills 是 Agent 可以使用的外部能力。

Skill 是更产品化的说法：它是一个做单件事的能力单元，有明确输入、输出、权限和风险等级。Tool / Function 是它在工程里的调用形态。

常见工具类型：

| 类型 | 示例 | 风险 |
| --- | --- | --- |
| 只读工具 | search_kb、query_order、get_weather | 低到中 |
| 写操作工具 | send_message、create_ticket、change_address | 中到高 |
| 执行工具 | run_sandbox_code、execute_workflow | 高 |

灵魂原则：

```text
工具是边界，不是功能列表。
```

工具白名单、参数校验、权限控制，是 Agent 的安全生命线。

不是所有 Skill 都必须藏在 Agent 后面。有些低风险、可校验、无副作用的 Skill 可以通过按钮、指令、表单给用户直通使用。

但有副作用的 Skill，例如发送消息、修改数据、创建订单，必须放在 Agent/Workflow 后面，并经过预览、确认和审计。

### C. 记事本：Context / Memory

Agent 不能真正无限记住所有信息，它靠上下文和结构化状态管理任务。

常见内容：

- 系统提示。
- 对话历史。
- 工具返回结果。
- 已确认字段。
- 关键状态。
- 拒绝原因。
- 用户确认记录。

Memory 的难点不是越长越好，而是：

- 什么时候压缩。
- 什么信息绝不能丢。
- 什么信息不能跨会话保留。
- 什么信息需要脱敏。

例如订单 Agent 中，订单号、权限结论、用户确认状态不能丢。

### D. 刹车与方向盘：Controller / Harness

Controller 或 Harness 是 Agent 的调度器和边界控制层。

它规定：

- 最大步数。
- 单步超时。
- 总任务超时。
- 工具白名单。
- 工具调用前权限检查。
- 写操作前人工确认。
- 什么时候算完成。
- 什么时候强制停止。
- 工具返回异常怎么办。
- 是否允许用户中途打断。

没有 Controller 的 Agent 很容易变成无限循环、误调用或成本失控。

### E. 日志与审计：Trace

Trace 记录每一次「想、调工具、拿结果、再决定」。

至少记录：

- Step 编号。
- LLM 决策。
- 调用工具。
- 工具参数。
- 工具返回。
- 延迟。
- 错误。
- 用户确认。
- 最终输出。

示例：

```text
Step1 thought=查物流, tool=query_order, params={order_id:123}, result=已发货, latency=320ms
Step2 thought=需要生成短信预览, tool=none, result=preview_message
Step3 user_confirm=true
Step4 tool=send_message, result=success
```

Trace 的价值是：出事后能复盘，而不是变成「模型自己说的」。

## 4. Agent 不等于多轮聊天

| 多轮聊天 | Agent |
| --- | --- |
| 只是来回聊 | 聊天 + 调用工具/系统 |
| 不一定触达外部世界 | 会查数据库、调用接口、创建记录 |
| 错了通常只是体验问题 | 错了可能是业务风险 |
| 不一定需要审计 | 必须可审计 |

只要 AI 触达外部系统、数据库、消息发送、工单、审批或数据修改，就应该按 Agent 来治理，哪怕它只执行一两步。

## 5. Agent 的两种常见产品形态

### 只查不改的知识 Agent

能力：

- 查知识库。
- 查订单。
- 查制度。
- 查报表。
- 总结和回答。

重点：

- 权限控制。
- 引用来源。
- 拒答和澄清。
- 检索质量门。
- Trace。

### 查 + 改/建单的任务 Agent

能力：

- 查询信息。
- 创建工单。
- 发送消息。
- 修改地址。
- 提交审批。

重点：

- 工具白名单。
- 写操作预览。
- 二次确认。
- 可撤回。
- 审计日志。
- 人工接管。

## 5.1 真实业务案例：订单安抚 Agent

用户说：

```text
帮我查订单 #123 的物流，然后给客户发一条安抚短信。
```

系统应该这样走：

1. Gateway 判断这是「查询 + 消息发送」任务，进入 Agent 链路。
2. Agent 调用 `query_order` 查询物流。
3. 工具返回：已发货，预计明天送达。
4. Agent 生成短信预览。
5. 用户确认。
6. Agent 调用 `send_message_template` 发送模板化消息。
7. Trace 记录查询、预览、确认和发送结果。

关键点：

- 查询订单是只读 Skill，可以自动执行。
- 发送短信是有副作用 Skill，必须预览和确认。
- 如果用户没有权限查看该订单，流程应在工具或 Harness 层被拦截。

## 6. Agent 与 Skill 的关系

```text
Skill = 做一件事的能力单元
Agent = 决定先调用哪个 Skill、后调用哪个 Skill 的调度者
```

例如：

- `query_order` 是一个 Skill。
- `send_message` 是一个 Skill。
- `create_ticket` 是一个 Skill。
- 订单处理 Agent 会根据任务状态决定先查订单，再生成回复，再等待用户确认，再发送消息或建单。

用户也可以直接使用某些 Skill，但必须是产品包装后的受控入口：

- `/订单状态`。
- 「查物流」按钮。
- 表单提交。
- 预览卡里的「确认执行」按钮。

产品上要为每个 Skill 定义：

- 是否允许 Agent 调用。
- 是否允许用户直通。
- 是否有副作用。
- 是否需要人工确认。
- 是否需要审计。

## 7. AI PM 验收 Agent 的 6 个硬问题

### 1. 白名单定义了吗？

除了清单里的工具，任何工具调用都应该被拦截。

### 2. 写操作用户确认了吗？

发送消息、创建任务、修改数据、提交审批，都不能只靠模型礼貌询问，必须由产品流程硬控。

### 3. 停得下来吗？

必须设置：

- 最大步数。
- 单步超时。
- 总任务超时。
- 最大成本。
- 循环检测。

### 4. 权限在哪里校验？

推荐两层校验：

- Harness 前置校验。
- 工具内部再次校验。

### 5. 工具返回垃圾怎么办？

要定义 fallback：

- 重试。
- 降级。
- 拒答。
- 澄清。
- 转人工。
- 记录 case。

### 6. 能复盘吗？

Trace 里必须能还原：

- 哪一步开始跑偏。
- 调了什么工具。
- 参数是什么。
- 工具返回什么。
- 用户是否确认。
- 为什么最终输出。

## 7.1 反例：没有 Harness 的 Agent 会怎样翻车

错误做法：

```text
让模型自己决定要不要查订单、要不要发短信、要不要继续尝试。
```

可能后果：

- 模型把查地址误判成改地址。
- 模型在未确认时发送消息。
- 工具调用失败后继续编造结果。
- 循环多次重试导致成本失控。
- 出事后没有 Trace，无法复盘责任。

Agent 的风险不是它不够聪明，而是它在错误边界里继续执行。

## 8. PRD 里应该写什么

Agent PRD 至少要包含：

- Agent 目标。
- 适用场景。
- 不适用场景。
- 工具白名单。
- Skill 暴露策略。
- 工具输入输出。
- 工具权限。
- 状态字段。
- 最大步数和超时。
- 人工确认节点。
- 中断和回滚。
- Trace 字段。
- Eval 指标。
- 风险分级。

## 8.1 PRD 可粘贴写法

```text
本 Agent 采用受控循环执行。
Agent 仅可调用工具白名单中的 Skill。
只读 Skill 可自动执行；写操作 Skill 必须先生成预览，并在用户确认后执行。
Agent 单次任务最大执行 5 步，总时长不超过 120 秒。
每一步工具调用、参数、返回、用户确认和最终结果均写入 Trace。
```

工具白名单：

| Skill | 类型 | 风险 | 是否自动执行 | 是否需确认 |
| --- | --- | --- | --- | --- |
| query_order | 只读 | 低 | 是 | 否 |
| send_message_template | 消息 | 中/高 | 否 | 是 |
| create_ticket | 写数据 | 中/高 | 否 | 是 |

## 9. 常见事故模式

- 没有工具白名单，模型尝试调用不该调用的能力。
- 把高风险 Skill 直接暴露给用户。
- 写操作没有确认，导致误发、误改、误建单。
- 没有停止条件，导致无限循环和成本爆炸。
- 权限只在 Prompt 里写，没有系统级校验。
- 工具返回错误后，模型基于错误信息继续编。
- 没有 Trace，事故无法复盘。
- 目标太大，Agent 在多步任务中不断偏航。

## 10. 人话总结

Agent 不是有性格的 AI。

它是一个受控循环：

- LLM 负责决定下一步。
- Tool/Skill 负责触达外部世界。
- Memory 负责保存任务状态。
- Harness 负责限制能碰什么、跑多远、什么时候刹车。
- Trace 负责让所有动作可审计。

没有工具白名单、没有停止条件、没有确认机制的 Agent，不是产品能力，而是事故预演。

## 11. 一句可复述总结

Agent 的价值不是让 AI 自由发挥，而是让 AI 在受控边界里一步步做事；边界、刹车、确认和审计，才是 Agent 产品化的核心。
