# Skill 能力单元与暴露策略

## 0. 一句话抓住本质

Skill 是一个动词的实现体，Agent 是决定什么时候调用这些动词的调度者。

## 1. 一句话定义

Skill 是系统级的可调用能力单元。它做一件明确的事，有清晰输入、输出、权限和风险等级。

Agent 是调度者，决定先调用哪个 Skill、后调用哪个 Skill。

用户直接使用 Skill，本质上是产品把某个安全、单一、可校验的能力包装成按钮、指令或表单入口。

## 2. Skill 更像汽车里的什么

如果 LLM 是发动机，Agent 是驾驶员加行车电脑，Skill 更像车载系统里的附件功能：

- 雨刮器。
- 空调。
- 电动窗。
- 充气泵。
- 绞盘。
- 外放电插座。

它们的共同点是：

- 做一件事，做完就停。
- 有明确输入和输出。
- 可以被用户按钮触发。
- 也可以被自动驾驶/辅助驾驶逻辑触发。
- 有些可以直接暴露，有些必须锁在系统内部。

所以：

```text
Agent : Skill = 调度者 : 能力单元
```

## 3. Skill 的架构视角

工程上，Skill 通常会被设计成 Tool / Function。

示例：

```json
{
  "name": "query_order",
  "description": "查询订单状态，只读",
  "parameters": {
    "order_id": "string"
  },
  "risk": "low",
  "allow_user_direct": true
}
```

这意味着：

- Agent 可以调用它。
- Router 可以在意图确定时直通调用它。
- UI 可以把它包装成按钮或表单。
- Harness 可以根据风险决定是否允许直通。

Skill 最好一开始就按工具规格设计，否则未来 Agent 很难接住。

## 4. Agent 调用 vs 用户直通

### Agent 路径

```text
用户目标
→ LLM 规划
→ 选择 Skill
→ 带参数调用
→ 获取结果
→ 继续规划或最终回复
```

适合：

- 多步骤任务。
- 需要条件判断。
- 需要多个 Skill 串联。
- 需要工具结果影响下一步。

### 用户直通路径

```text
用户点击按钮/输入指令/提交表单
→ Router 确认意图
→ 参数校验
→ 调用 Skill
→ 渲染结果
```

适合：

- 单一动作。
- 意图确定。
- 低风险。
- 参数可校验。
- 无副作用或可撤销。

用户直通不是让用户直接碰工具底层，而是产品给了一个受控入口。

## 5. 用户直接用 Skill 的产品形态

| 产品形态 | 本质 | 示例 |
| --- | --- | --- |
| 斜杠命令 | 路由直通 Skill | `/订单状态` → 查询订单 |
| 按钮 | 路由直通 Skill | 「查物流」按钮 |
| 表单提交 | 参数来自表单 | 填订单号后查询 |
| 预览卡确认 | Agent 预览后调用 Skill | 改地址前点确认 |

共同点：

- 入口是产品给的。
- 参数经过清洗和校验。
- Skill 在白名单里。
- 风险等级已判断。
- 必要时有确认和审计。

## 6. Skill 暴露策略

| Skill 类型 | 汽车等价 | 是否允许用户直通 | 暴露方式 | 控制措施 |
| --- | --- | --- | --- | --- |
| 只读查询 | 雨刮器/空调读数 | 允许 | 按钮、指令、表单 | 权限校验、日志 |
| 生成类 | 电动窗 | 允许 | 生成卡片、编辑、采纳 | Schema、禁用词、反馈 |
| 通知/消息 | 喇叭/转向灯 | 谨慎允许 | 模板化预览 | 模板、预览、确认 |
| 写数据 | 绞盘/差速锁 | 不允许直通 | Agent 预览后确认 | 二次确认、审计、可撤回 |
| 内部工具 | 高压维修接头 | 绝不暴露 | 内部 Harness 使用 | 内部权限、审计 |

一句话：

```text
能直通的 Skill 必须低风险、可校验、无副作用或可撤销。
有副作用的 Skill 必须藏在 Agent/Workflow 后面，并装刹车。
```

## 6.1 真实业务案例：订单平台的 Skill

假设订单平台有 5 个 Skill：

| Skill | 类型 | 是否可直通 | 原因 |
| --- | --- | --- | --- |
| query_order | 只读查询 | 可以 | 低风险，参数可校验 |
| generate_reply | 生成类 | 可以 | 输出可编辑和采纳 |
| send_message | 消息 | 谨慎 | 必须模板化和确认 |
| change_address | 写数据 | 不可以 | 有副作用，必须预览和审计 |
| read_system_log | 内部工具 | 不可以 | 高权限内部能力 |

用户点击「查物流」，可以直通 `query_order`。

但用户说「帮我改地址并通知客户」，必须进入 Agent/Workflow：

```text
查订单 → 生成改地址预览 → 用户确认 → 修改地址 → 生成短信预览 → 用户确认 → 发送
```

同样是 Skill，暴露方式完全不同。

## 6.2 反例：把所有 Skill 都给用户直通

错误做法：

```text
把 query_order、send_message、change_address、read_system_log 都做成按钮或指令。
```

后果：

- 用户误点导致数据被修改。
- 高权限工具被普通用户看到。
- 缺少预览和确认，消息误发。
- 没有审计，事故难以追责。

Skill 暴露策略不是 UI 选择，而是风险治理。

## 7. PM 要定义的字段

每个 Skill 至少要定义：

| 字段 | 说明 |
| --- | --- |
| name | Skill 名称 |
| description | 做什么 |
| input_schema | 输入字段和约束 |
| output_schema | 输出字段和格式 |
| risk_level | 风险等级 |
| side_effect | 是否有副作用 |
| reversible | 是否可撤回 |
| permission_required | 权限要求 |
| allow_user_direct | 是否允许用户直通 |
| require_confirmation | 是否需要确认 |
| audit_required | 是否需要审计 |
| owner | 负责人 |

## 8. Skill、Tool、Agent、Workflow 的区别

| 概念 | 解释 |
| --- | --- |
| Skill | 做一件事的能力单元 |
| Tool / Function | Skill 在工程里的调用形态 |
| Agent | 决定调用哪些 Skill，以及调用顺序的调度者 |
| Workflow | 预设流程，按固定或半固定步骤编排 Skill |
| Harness | 控制 Skill/Agent 能做什么、如何确认、如何审计的外壳 |

## 9. 常见误区

- 以为 Skill 只能给 Agent 用。
- 以为用户直通 Skill 就是把工具列表暴露给用户。
- 所有 Skill 都做成自由聊天入口。
- 写操作 Skill 允许直通。
- 没有给 Skill 标风险等级。
- 没有定义输入输出 Schema。
- 没有权限、确认和审计。

## 9.1 PRD 可粘贴写法

```text
所有 Skill 必须登记风险等级、输入输出 Schema、副作用、是否可撤回、是否允许用户直通。
低风险只读 Skill 可以通过按钮、指令或表单直通。
有副作用 Skill 不允许直通，必须通过 Agent/Workflow 执行，并要求预览、确认和审计。
内部工具不对普通用户外显。
```

## 10. 人话总结

Skill 是动词的实现体。

Agent 是决定先干哪件、后干哪件的调度者。

用户直接用 Skill，是把某个安全、单一、可校验的动词包装成按钮、指令或表单给用户按。

但按钮背后仍然是受控系统调用，不是把底层工具交给用户随便用。

## 11. 一句可复述总结

Skill 可以被 Agent 调，也可以被用户通过按钮直通；但能不能直通，不看技术能不能做，而看风险、权限、副作用和可撤回性。
