# Prompt 工程与工作流设计

## 1. 这个知识点是什么

Prompt 是给 AI 的任务指令。Prompt 工程不是写一句神奇咒语，而是把任务目标、输入、规则、示例和输出格式组织清楚。

更准确地说，Prompt 是你对模型行为的软约束界面。

工作流设计则是把一个复杂任务拆成多个 AI 可执行步骤。

## 2. 为什么重要

Prompt 是 AI 产品最快的验证方式。

在产品早期，你可以不写代码，只通过 Prompt 验证：

- 模型是否能理解任务。
- 输出是否有价值。
- 哪些输入会导致失败。
- 是否需要知识库。
- 是否需要人工审核。

## 3. 一个好 Prompt 的结构

### 角色

告诉模型它应该以什么身份工作。

例如：你是一名资深用户研究员，擅长从访谈记录中提取用户目标、痛点和需求机会。

### 任务

明确要做什么。

例如：请分析以下访谈记录，并输出用户痛点、行为动机、阻碍因素和产品机会点。

### 输入

说明输入内容是什么。

例如：输入是一段用户访谈文本，可能包含主持人问题和用户回答。

### 规则

说明限制条件。

例如：

- 只能基于原文分析。
- 不要编造用户没有说过的内容。
- 每条洞察都要附原文证据。

### 输出格式

指定输出结构。

例如：

```json
{
  "insights": [
    {
      "theme": "string",
      "pain_point": "string",
      "evidence": "string",
      "opportunity": "string"
    }
  ]
}
```

在真实产品中，输出格式最好进一步变成 Schema 契约。也就是明确字段名、类型、长度、枚举、失败处理，让模型输出能被系统稳定解析。

### 示例

给模型一个输入和输出示例，可以显著提升稳定性。

## 4. System Prompt、Context、User Input

生产环境里的 Prompt 往往由三层组成：

- System Prompt：定义角色、铁律、禁止项、输出规则。
- Context：来自知识库、业务规则、商品库、历史记录等上下文材料。
- User Input：用户这次输入的动态变量。

可以理解为：

```text
最终 Prompt = System Prompt + Context + User Input
```

AI PM 要把这三层写清楚，而不是只写一句「帮用户生成内容」。

## 5. 工作流拆解

复杂任务不要一次让模型完成。

例如「生成用户研究报告」可以拆成：

1. 清洗访谈文本。
2. 提取用户原话。
3. 识别痛点。
4. 合并相似痛点。
5. 生成洞察主题。
6. 生成机会点。
7. 输出报告。
8. 检查引用是否来自原文。

## 6. 常见 Prompt 类型

- 信息抽取 Prompt。
- 分类 Prompt。
- 总结 Prompt。
- 改写 Prompt。
- 生成方案 Prompt。
- 评审 Prompt。
- 自检 Prompt。
- 路由 Prompt。

## 7. 工作中怎么用

你可以先用 Prompt 做低成本原型：

- 找 5-10 条真实材料。
- 写一个初版 Prompt。
- 观察输出问题。
- 修改任务说明、规则和格式。
- 记录错误案例。
- 沉淀为 Prompt 版本。

## 8. 可交付物

- Prompt 设计文档。
- Prompt 版本记录。
- 输入输出样例。
- 错误案例清单。
- Prompt 评测报告。
- Output Schema 契约。
- Prompt/Eval diff 记录。

## 9. 常见坑

- 只写一句模糊指令。
- 不给输出格式。
- 不提供示例。
- 把多个复杂任务塞进一个 Prompt。
- 没有记录版本，导致效果变化无法追踪。
- System Prompt 散落在代码里，无法评审和回滚。
- 没有 Schema，导致前端和后端只能猜模型输出。
