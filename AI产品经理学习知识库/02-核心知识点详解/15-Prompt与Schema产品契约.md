# Prompt 与 Schema 产品契约

## 0. 一句话抓住本质

Prompt 是你给模型的任务说明，Schema 是你和模型约定的输出接口。

Prompt 让模型知道「怎么做」，Schema 让产品系统知道「怎么接」。

## 1. 这个知识点是什么

Prompt 是你给模型的任务说明，Schema 是你规定模型必须返回的结构。

如果说 Prompt 是任务说明和行为约束，Schema 就是模型输出的接口契约。

传统产品里，前端调用接口后拿到稳定字段。AI 产品里，模型也必须尽量返回稳定字段，否则前端、后端、审核和数据统计都会变得不可控。

## 2. 为什么重要

只让模型「写一段话」，会导致：

- 输出格式不稳定。
- 前端无法稳定解析。
- 数据库无法稳定存储。
- 审核逻辑无法自动检查。
- 评测难以标准化。
- 页面可能因为多余文字或字段缺失而出错。

Schema 的价值，是把模型输出从自然语言散文变成可用的产品零件。

## 2.1 用汽车比喻讲明白

如果 LLM 是发动机，Prompt 像驾驶指令，Schema 像车辆接口标准。

你不能只对车说「开得好一点」。你要说明：

- 去哪里。
- 走什么规则。
- 不能超速。
- 到达后给我什么状态信息。

同样，你不能只让模型「帮我生成」。你要规定：

- 任务是什么。
- 材料从哪里来。
- 禁止做什么。
- 输出字段是什么。
- 字段不合规怎么办。

没有 Schema 的 AI 输出，就像一个没有标准接口的零件，看起来能用，但很难稳定装进产品系统。

## 3. Prompt 的三层拼装

生产环境中的 Prompt 通常不是一句话，而是三层拼装：

### System Prompt

定义角色、任务边界、铁律和禁止项。

例如：

```text
你是商品营销文案助手。
必须遵守：
- 不得编造商品参数。
- 不得出现价格承诺。
- 输出必须符合指定 JSON Schema。
```

System Prompt 应该版本化管理。改一次 System Prompt，本质上就是改一次 AI 策略。

### Context

Context 是模型这次可以依据的材料。

来源可能包括：

- RAG 检索结果。
- 商品库字段。
- 用户资料。
- 历史对话。
- 业务规则。
- 禁用词库。

### User Input

User Input 是用户这次操作带来的动态变量。

例如：

- 风格偏好。
- 补充关键词。
- 文档内容。
- 任务目标。

最终 Prompt 可以理解为：

```text
final_prompt = System Prompt + Context + User Input
```

## 4. Schema 是什么

Schema 是输出结构约定。

你可以把它理解成一张固定表单：

- 字段名。
- 字段类型。
- 是否必填。
- 长度限制。
- 枚举值。
- 数组数量。
- 失败处理。

例如：

```json
{
  "headline": "string, <=20字",
  "body": "string, <=60字",
  "tags": ["string"],
  "tone": "干练|温柔|活力"
}
```

## 5. Schema 在 PRD 里怎么写

建议用表格写成契约：

| 字段 | 类型 | 约束 | 示例 | 失败处理 |
| --- | --- | --- | --- | --- |
| headline | string | <=20字；禁止价格承诺 | 轻量牛皮通勤包 | 超长则重生成 |
| body | string | <=60字；只能使用给定信息 | 头层牛皮，A4轻松收纳 | 含禁用词则拦截 |
| tags | string[] | 最多3个；从词库选择 | 通勤、轻量 | 不在词库则过滤 |
| tone | enum | 干练/温柔/活力 | 干练 | 不匹配则重试 |

## 6. AI 生成链路

典型链路如下：

```text
用户点击生成
→ 前端收集表单字段
→ 后端拼 Prompt
→ 调用模型并约束输出 Schema
→ 解析 JSON
→ 校验字段
→ 通过则展示
→ 不通过则重试、降级或提示补充信息
```

## 6.1 真实业务案例：用户研究洞察卡片

用户上传一段访谈记录，希望 AI 生成洞察卡片。

错误写法：

```text
请帮我总结这段访谈，提炼用户洞察。
```

这个 Prompt 会导致：

- 洞察数量不稳定。
- 有时没有证据。
- 有时把主持人观点当成用户观点。
- 前端无法稳定渲染卡片。

更好的写法是把 Prompt 和 Schema 合在一起：

```text
你是资深用户研究员。
只能基于访谈原文提炼洞察。
每条洞察必须附原文证据。
如果证据不足，不要生成该洞察。

请只输出 JSON。
```

输出 Schema：

| 字段 | 类型 | 约束 | 失败处理 |
| --- | --- | --- | --- |
| title | string | <= 20 字 | 超长则重生成 |
| insight | string | 必须基于原文 | 无证据则删除 |
| evidence | string | 必须是原文片段 | 不匹配则标记失败 |
| confidence | enum | 高/中/低 | 不合法则重试 |
| opportunity | string | 可为空 | 不确定则填「待确认」 |

这样前端可以把每条洞察稳定渲染成卡片，评测也可以检查字段是否完整、证据是否准确。

## 6.2 反例：只靠 Prompt 不靠 Schema 的翻车点

常见翻车：

- 模型在 JSON 前面加「好的，以下是结果」。
- 模型把数组输出成自然段。
- 模型漏掉必填字段。
- 模型输出了页面放不下的长文本。
- 模型把禁用词写进结果。
- 模型输出无法解析，前端页面空白。

这类问题不是模型不够聪明，而是产品没有把输出契约定义清楚。

## 7. 质量护栏

PM 要盯住：

- System Prompt 是否有版本号。
- Schema 是否能被前后端理解。
- 模型输出是否能通过 JSON 解析。
- 字段是否完整。
- 枚举是否越界。
- 长度是否超限。
- 禁用词是否命中。
- 解析失败时是否有兜底。

## 8. 工作中怎么用

写 AI PRD 时，不要只写「AI 生成文案」。

要写：

- Prompt 由哪几层组成。
- 每层内容来自哪里。
- 哪些变量是动态的。
- 输出 Schema 是什么。
- 校验不过怎么办。
- 哪些错误要进入日志。

## 8.1 PRD 可粘贴写法

```text
本功能采用 System Prompt + Context + User Input 三层 Prompt 组装。
模型输出必须符合 Output Schema。
后端在进入渲染层前执行 JSON 解析和字段校验。
若解析失败，自动重试一次；仍失败则展示降级提示，并记录 error_code。
```

## 8.2 验收问题

- System Prompt 是否有版本号？
- Context 来源是否明确？
- 用户输入是否经过清洗？
- 输出是否必须是裸 JSON？
- 必填字段缺失怎么办？
- 枚举越界怎么办？
- 字段超长怎么办？
- 解析失败是否有重试和 fallback？
- Prompt 变更后是否跑 Eval diff？

## 9. 可交付物

- Prompt 组装说明。
- System Prompt 版本记录。
- Output Schema 契约。
- 字段校验规则。
- 解析失败处理方案。
- Prompt/Eval diff 记录。

## 10. 一句可复述总结

Prompt 决定模型怎么想，Schema 决定产品怎么接。没有 Schema，AI 输出只是文本；有了 Schema，AI 输出才是产品零件。
