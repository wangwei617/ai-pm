# Agent 与工具调用

## 1. 这个知识点是什么

Agent 是一种能围绕目标进行多步执行的 AI 产品形态。它不只是回答问题，还可以调用工具、查询资料、处理文件、执行操作，并在过程中根据结果调整下一步。

工具调用是 Agent 的关键能力。工具可以是搜索、数据库查询、发邮件、生成文档、创建工单、调用业务接口等。

更产品化地说：

```text
Agent = 在受控循环里运行的 LLM + Tools + Memory + Controller/Harness + Trace
```

普通 LLM 调用是一次性的：

```text
输入 → 一次生成 → 输出
```

Agent 是循环式的：

```text
输入 → 想下一步 → 调工具 → 拿结果 → 再想 → 最终输出/行动
```

所以 Agent 的本质不是「更聪明」，而是多了工具、循环、停止条件、权限边界和可追溯日志。

## 2. 为什么重要

早期 AI 产品大多是「问答」。但真实工作往往不是问答，而是一串动作：

- 查资料。
- 分析信息。
- 做判断。
- 生成内容。
- 调用系统。
- 等待确认。
- 完成提交。

Agent 让 AI 从「会说」变成「能做」。

## 3. Agent 的基本组成

### 目标

Agent 必须有明确目标。

例如：根据用户访谈记录生成一份包含洞察、证据和产品机会点的研究报告。

### 工具

工具是 Agent 可以调用的能力。

例如：

- 文件读取工具。
- 知识库检索工具。
- 数据库查询工具。
- 文档生成工具。
- 邮件发送工具。
- 工单创建工具。

工具不是越多越好。工具白名单、参数校验和权限控制，是 Agent 的安全边界。

### 计划

Agent 需要把目标拆成步骤。

例如：

1. 读取材料。
2. 提取关键信息。
3. 检索相关知识。
4. 生成洞察。
5. 检查证据。
6. 输出报告。

### 状态

状态记录 Agent 当前执行到哪里、已经拿到什么结果。

状态可以包括：

- 已确认字段。
- 工具返回结果。
- 用户确认状态。
- 拒绝原因。
- 当前步骤。
- 是否达到完成条件。

### Controller / Harness

Controller 或 Harness 决定 Agent 能跑多远、能碰什么、什么时候停。

需要定义：

- 最大步数。
- 单步超时。
- 总任务超时。
- 工具白名单。
- 写操作确认。
- 权限校验。
- 失败降级。
- 中断和回滚。

### 人工确认

高风险动作必须让用户确认。

例如：

- 发送邮件。
- 提交审批。
- 修改数据库。
- 删除内容。
- 向客户发送消息。

### Trace

Trace 是 Agent 的执行日志，用于复盘每一步发生了什么。

至少记录：

- Agent 决策。
- 工具名称。
- 工具参数。
- 工具返回。
- 延迟。
- 错误。
- 人工确认记录。

## 4. 适合 Agent 的场景

- 多步骤任务。
- 需要调用多个工具。
- 需要跨系统完成工作。
- 每一步结果会影响下一步。
- 用户希望 AI 代办，而不是只给建议。

例如：

- AI 销售跟进助手。
- AI 客服工单处理助手。
- AI 招聘筛选助手。
- AI 数据分析助手。
- AI 用户研究助手。

## 5. 不适合 Agent 的场景

- 单步问答。
- 简单文本生成。
- 高风险且不能人工确认的操作。
- 工具权限无法控制。
- 任务目标不清晰。

## 6. 工作中怎么用

设计 Agent 功能时，要写清楚：

- Agent 的目标是什么。
- 用户什么时候启动 Agent。
- Agent 有哪些可用工具。
- 每个工具的输入和输出是什么。
- 哪些动作需要用户确认。
- 最大步数是多少。
- 停止条件是什么。
- 权限在哪里校验。
- Agent 执行失败时怎么办。
- 是否需要执行日志。
- 是否允许用户中途打断。

## 7. 可交付物

- Agent 任务流程图。
- 工具调用清单。
- 工具权限说明。
- 人工确认规则。
- Agent 执行日志设计。
- Agent 失败处理方案。
- 工具白名单。
- Trace 字段定义。
- 停止条件和超时规则。

## 8. 常见坑

- 目标太大，Agent 无法稳定完成。
- 工具权限过宽，带来安全风险。
- 没有人工确认，高风险操作失控。
- 没有执行日志，失败后无法排查。
- 把简单功能强行做成 Agent。
- 没有最大步数，导致无限循环和成本失控。
- 权限只写在 Prompt 里，没有系统级校验。
- 工具返回异常后没有 fallback。
