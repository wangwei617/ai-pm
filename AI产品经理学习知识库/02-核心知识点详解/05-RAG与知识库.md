# RAG 与知识库

## 1. 这个知识点是什么

RAG 是 Retrieval-Augmented Generation 的缩写，意思是检索增强生成。

简单理解：先从知识库中检索和用户问题相关的内容，再把这些内容交给大语言模型生成答案。

## 2. 为什么重要

大语言模型本身不一定知道你的企业知识、产品规则、内部流程和最新资料。RAG 可以让 AI 基于指定知识回答问题，降低幻觉，提高专业性。

适合使用 RAG 的场景：

- 企业知识库问答。
- 客服助手。
- 法规政策查询。
- 产品文档问答。
- 销售资料助手。
- 用户研究资料分析。

## 3. RAG 的基本流程

1. 收集知识文档。
2. 清洗文档。
3. 将文档切分成片段。
4. 将片段转成向量。
5. 用户提问时检索相关片段。
6. 必要时进行过滤、重排和去重。
7. 将检索结果和问题一起发给模型。
8. 模型基于资料生成答案。
9. 展示答案和来源。

## 4. 核心概念

### Chunk

Chunk 是文档切分后的片段。

切得太大，检索不精准。切得太小，语义不完整。

AI PM 不一定要自己配置切分算法，但要知道切分会影响回答质量。

### Embedding

Embedding 是把文本转成向量，方便系统计算语义相似度。

它让系统能找到「意思相近」而不是「字面完全相同」的内容。

### 检索

检索是从知识库中找出最相关内容。

常见问题：

- 找不到相关内容。
- 找到了不相关内容。
- 找到的内容不完整。
- 多份文档互相冲突。

### Top-K 召回

Top-K 召回，是指系统从知识库里找出最相关的前 K 个知识片段。

你可以把它理解成：

```text
用户问问题 → 系统去资料库搜索 → 捞出最相关的前 K 段原文 → 准备塞进 Prompt
```

常见配置是 Top-3 或 Top-5。

- K 太小：可能漏掉关键上下文。
- K 太大：会把噪音塞进模型，让答案变差，也增加成本。

AI PM 要关注的不只是有没有召回，而是召回回来的内容是否足够靠谱。

### 召回质量门

召回质量门，是决定「是否允许模型基于检索结果生成答案」的判断规则。

例如：

```text
如果召回数量为 0，或最高相关度低于阈值：
  不进入生成
  走拒答或澄清
否则：
  把 chunk 塞进 Prompt
  生成带引用答案
```

这能避免「资料没找到，但模型硬编一个答案」。

### 重排

重排是对检索结果再次排序，让最相关内容排在前面。

复杂知识库通常需要重排来提高质量。

### 混合检索

混合检索是把向量语义检索和关键词检索结合起来。

它适合处理专有名词、编号、法规条款、产品型号等关键词非常重要的场景。

### 元数据过滤

元数据是文档的附加信息，比如部门、日期、权限、版本、来源。

产品上可以用元数据过滤：

- 过期内容。
- 用户无权限内容。
- 非目标部门内容。
- 低可信来源内容。

### 引用来源

引用来源是 RAG 产品建立信任的重要方式。

用户应该能看到 AI 的回答来自哪份文档、哪一段内容。

## 5. 工作中怎么用

设计 RAG 功能时，你要明确：

- 知识来源是什么。
- 谁负责维护知识。
- 文档多久更新一次。
- 是否需要权限控制。
- AI 回答是否必须引用来源。
- 找不到答案时怎么处理。
- 多个来源冲突时怎么处理。
- 是否需要重排、过滤、去重或冲突标注。
- Top-K 取多少。
- 相关度低于多少时拒答。
- 什么时候拒答，什么时候澄清。

## 6. 可交付物

- 知识库范围说明。
- 文档清洗规则。
- RAG 产品流程图。
- 引用展示方案。
- 知识库维护方案。
- RAG 评测集。
- 检索质检报告。
- RAG 上线验收清单。

## 7. 常见坑

- 以为接入知识库就不会幻觉。
- 文档质量差，导致答案质量差。
- 不展示来源，用户无法信任。
- 没有权限控制，可能泄露敏感知识。
- 知识库长期不更新。
- 找不到答案时仍然让模型硬答。
- 不看检索返回的 chunk，只看最终答案。
- 改了 chunk 或 rerank 后不跑回归评测。
- 没有设置召回质量门，导致低质量资料也被塞给模型。
- 把拒答当成失败，而不是保护用户体验。
