# Fine-tuning 与领域适配

## 1. 这个知识点是什么

Fine-tuning，也就是微调，是在已有大模型基础上，用特定领域或任务数据继续训练，让模型更稳定地表现出某类行为。

领域适配是更大的概念，除了微调，还包括 Prompt、RAG、规则、工具调用、知识库、反馈闭环等手段。

## 2. 为什么重要

很多团队一遇到模型效果不好，就想微调。但对 AI PM 来说，更重要的问题是：这个问题是否真的应该用微调解决。

在多数业务场景里，优先级通常是：

```text
Prompt 优化 → 输出 Schema → RAG → 工具调用/规则校验 → 微调
```

微调不是万能按钮。它更适合固化一类稳定行为，而不是修补产品链路里的所有问题。

## 3. 微调适合解决什么

### 固定风格

例如客服语气、品牌文案风格、内部报告风格。

### 稳定分类

例如把用户反馈分成固定标签，把工单分成固定类型。

### 特定格式输出

例如长期要求模型输出某类结构化内容。

### 领域表达

例如法律、医疗、金融、企业内部术语，但前提是数据质量高且任务边界稳定。

## 4. 微调不适合解决什么

- 缺少最新知识。
- 需要实时查询数据库。
- 需要严格权限控制。
- 需要引用来源。
- 复杂业务逻辑。
- 多步骤工具执行。
- 输入数据质量差。

这些问题通常应该优先考虑 RAG、工具调用、规则校验或产品流程设计。

## 5. 微调决策树

你可以按下面顺序判断：

1. 只改 Prompt 能不能解决？
2. 加输出 Schema 能不能稳定格式？
3. 加 RAG 能不能补充知识？
4. 加规则或工具校验能不能提高可靠性？
5. 是否有足够高质量标注数据？
6. 任务是否长期稳定？
7. 微调后的维护成本是否可接受？
8. 是否有评测集验证微调收益？

只有当以上问题都能回答清楚时，才值得进入微调。

## 6. 微调需要准备什么

### 数据

通常需要高质量输入输出样本。

PM 要关注：

- 样本是否覆盖真实场景。
- 标注标准是否一致。
- 是否包含边界样本。
- 是否有错误样本和反例。
- 是否会泄露敏感数据。

### 评测集

训练数据和评测数据必须分开。

否则模型可能只是背了训练数据，而不是学会了能力。

### 回归机制

微调可能提升某些场景，同时损害其他场景。

因此每次微调后都要跑回归测试。

## 7. 微调的风险

- 标注成本高。
- 维护成本高。
- 任务变化后需要重新调。
- 可能过拟合。
- 可能引入新偏差。
- 模型升级后效果需要重新验证。

## 8. 工作中怎么用

当业务方说「我们要训练自己的模型」时，你可以先问：

- 当前效果差在哪里？
- 是知识不足、格式不稳、语气不对，还是逻辑错误？
- 有没有样本证明？
- Prompt 和 RAG 是否已经尝试过？
- 是否有足够标注数据？
- 微调成功后如何评测？
- 失败后是否有替代方案？

## 9. 可交付物

- 微调可行性分析。
- 领域适配方案对比。
- 标注数据规范。
- 训练/评测数据划分说明。
- 微调评测报告。
- 回归测试报告。
