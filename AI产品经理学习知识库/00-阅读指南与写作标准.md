# 阅读指南与写作标准

这套知识库的目标不是堆 AI 名词，而是帮助你成长为 AI Product Owner。

判断一篇文档是否合格，不看它讲了多少概念，而看它是否能让你做到三件事：

- 听懂：能用人话解释给老板、研发和业务方。
- 会写：能写进 PRD、评审材料和作品集。
- 能用：能在真实项目里判断边界、风险和结果。

## 1. 每篇核心文档的标准结构

核心专题建议按下面结构阅读和补充：

```text
1. 一句话抓住本质
2. 用类比讲明白
3. 为什么 AI Product Owner 必须管
4. 真实业务案例
5. 错误做法和翻车点
6. PRD 写法或决策表
7. 验收问题
8. 一句可复述总结
```

这也是后续继续扩展知识库时的写作标准。

## 2. 本知识库的主案例

为了避免每个概念都悬空，建议用三个主案例贯穿理解。

### 案例 A：AI 用户研究洞察助手

适合解释：

- Prompt + Schema。
- 信息抽取。
- 证据引用。
- Eval。
- 数据飞轮。
- AI Product Owner 作品集。

### 案例 B：企业知识库/HR 制度助手

适合解释：

- RAG。
- Top-K 召回质量门。
- 拒答和澄清。
- 权限过滤。
- 引用准确率。

### 案例 C：Manus-like 环境操作型 Agent

适合解释：

- Gateway 路由。
- Agent Loop。
- Skill 暴露策略。
- 工具白名单。
- 高安全 Harness。
- Trace 和审计。

## 3. 读者要形成的核心判断

你读完后应该能自然说出这些话：

- Prompt 不是咒语，是模型行为的软约束界面。
- Schema 不是技术细节，是模型输出和产品系统之间的接口契约。
- RAG 的质量下限由召回质量和引用透明度决定。
- Agent 不是更会聊天的 AI，而是受控循环里的工具调用系统。
- Skill 是动词的实现体，Agent 是调度者。
- 路由不能写死在 Prompt 里，它是 Gateway 的确定性产品逻辑。
- Loop 不是搜索，而是 Agent 的执行回合。
- Eval 不是测试环节，而是 AI 产品本身的一部分。
- AI Product Owner 不是负责功能上线，而是负责 AI 在业务里跑出结果。

## 4. 写作风格要求

好的 AI Product Owner 文档应该：

- 先讲判断，再讲概念。
- 先讲场景，再讲技术。
- 先讲风险，再讲能力。
- 用表格表达边界和决策。
- 用案例解释抽象词。
- 用反例提醒常见坑。
- 最后给一句可以在会上复述的话。

不推荐：

- 只堆术语。
- 只写工具教程。
- 只写 happy path。
- 只写「AI 可以做什么」，不写「AI 不该做什么」。
- 只写功能，不写评测、风险和业务结果。
