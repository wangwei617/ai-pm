# AI 产品经理需要懂的 AI 知识（分层次 · 重实操）

核心原则先说清：**AI PM 不需要会手推梯度下降，但必须对「AI 能做什么 / 不能做什么 / 为什么翻车 / 怎么让它在产品里可控地跑起来」有清晰的边界感。** 你是在做**决策与取舍**，不是在写训练脚本。

---

## 一、地基层：AI 基本常识（必须过关）

这些是"不说外行话"的最低门槛：

| 概念 | 你要理解的要点（PM视角） |
|---|---|
| **模型 ≠ 算法** | 模型是"学出来的东西"，算法是学习方式；产品卖的是模型行为，不是算法论文 |
| **训练 / 微调 / 推理** | 训练=做模型（贵、慢）；推理=用模型（贵在并发/延迟/成本）；微调=为你的场景改模型 |
| **参数 / 超参数** | 不用算，但要明白：调参本质是"把业务目标翻译成模型学习约束" |
| **监督 / 无监督 / RL** | 知道什么时候需要标注数据、什么时候靠反馈信号、什么时候走强化学习路线 |
| **泛化 vs 过拟合** | 判断一个酷炫 demo 是"真能力"还是"背了训练集"的关键直觉 |
| **不确定性** | AI 输出本质是概率分布：你要设计"容错"，而不是假设它永远对 |

> ✅ 自检：你能用一句话向老板解释——**为什么同一个模型今天答对的明天可能答错**，而不被反问"那还能用？"

## 二、核心层：主流 AI 能力边界（你现在绕不开的就是这坨）

### 1) 大语言模型（LLM）到底"擅长什么 / 不擅长什么"
你需要能把抽象需求翻译成对 LLM 的合理定位：

- ✅ 擅长：语言理解与生成、摘要/改写/分类/提取、对话态交互、代码辅助、工作流编排（当 Agent）
- ⚠️ 不擅长/不可靠：精确计算、严格事实一致性、复杂多步逻辑推理、强时效数据（除非接外源）
- 🚫 不适合：把 LLM 当"确定性的规则引擎"硬塞（你会陷入疯狂 prompt 修补循环）

### 2) Prompt / System Prompt / Few-shot —— 本质是什么
- Prompt 不是魔法咒语，而是**你对模型行为的"软约束界面"**
- 理解：**上下文窗口（Context Window）**、**Token**、**温度/top-p** 对输出的影响趋势
- 关键能力：能把一个模糊需求写成**可复现的 prompt + 输入结构 + 输出 schema**

### 3) RAG（检索增强生成）——当前最主流的"落地形态"
你要能回答：
- 为什么需要 RAG（解决幻觉 + 私有知识 + 时效性）
- Embedding / 向量库的作用（相似召回）
- Chunk 切分策略如何影响效果
- "检索不准 → 生成瞎编"这条链怎么在产品侧缓解（重排/过滤/引用/拒答）

### 4) Fine-tuning / LoRA / 领域适配（知道什么时候该走这条路）
你要能判断：
- 纯 prompt + RAG 能否解决？（90%场景先走这条）
- 真需要微调时：数据量级、标注成本、回归风险、持续维护成本是否划算
- 微调 ≠ 万能按钮：它更像是"把一类稳定行为固化"，不是补逻辑漏洞

### 5) Agent / Tool-use / Workflow（越来越重要）
- 理解 Agent = LLM + 规划 + 工具调用 + 记忆 + 校验循环
- 知道它的失败模式：无限循环、工具滥用、信息泄露、成本爆炸
- 产品上核心不是"让它更聪明"，而是**给它更窄的动作空间 + 更多检查点**

## 三、数据与评估层（PM 最容易翻车的地方）

### 1) 数据观：AI 产品的竞争力往往在"数据飞轮"
- 标注体系怎么建（标签定义、歧义处理、SOP）
- 黄金集（Gold Standard）/ 基准测试集怎么维护
- 用户反馈如何回流（点赞/踩/修正/对话重写）→ 变成下一次迭代燃料

### 2) 评估指标：别只看"准不准"
你必须同时管两类指标：

| 类型 | 例子 | PM要盯的点 |
|---|---|---|
| 离线指标 | Accuracy / Recall / F1 / BLEU/ROUGE | 只能证明"实验室可能行"，不等于线上好 |
| 在线指标 | 采纳率、任务完成率、重试率、停留、人工二审通过率 | 更接近业务真相 |
| 体验信号 | 中断率、澄清次数、用户改 prompt 频率 | 暴露"模型逼用户迁就它" |

还要懂 **Eval as Product**：
- 红队测试（越权/敏感/注入攻击）
- 对抗 case 库（难样本回归）
- A/B 实验 + 分层发布（模型更新比传统功能更危险）

## 四、产品化工程与成本意识（决定你能不能"上线且活下来"）

这些是 PM 的"现实摩擦力"，不懂就会被研发牵着走：

- **Token / 调用次数 / 并发** → 成本模型怎么算（很多 AI 功能死在账单上）
- **延迟 SLA**：200ms / 800ms / 2s 对应完全不同的 UX 设计
- **部署形态**：云端 API vs 私有化 vs 端侧小模型（数据安全、合规、成本、体验互相拉扯）
- **内容安全**：涉政/黄赌毒/个人敏感信息（PII）+ prompt injection 防护
- **可观测性**：日志里能还原"哪一步检索错了 / 哪个 tool 返回异常 / 哪种用户输入导致退化"

## 五、AI 特有的"产品方法论"（这才是你拉开差距的地方）

| 传统 PM 习惯 | AI PM 要切换到的思维 |
|---|---|
| 需求 = 功能点 | 需求 = **任务边界 + 成功定义 + 容错机制** |
| 认为输出应该 deterministic | 设计 **不确定 UI**：置信度、引用来源、二次确认、可撤回、fallback |
| 验收靠"我觉得效果不错" | 验收靠 **case库 + 评分维度 + 回归集**（做 AI 质检台） |
| 把模型当黑盒不管 | 做到：能在高层解释失败归因（检索坏/指令弱/数据缺口/安全拦截） |

几个高频交付件建议你练熟：
- **能力说明书（What it does / When it fails / How we mitigate）**
- **Prompt/策略变更记录 + Eval diff**（任何 prompt 修改都要像发版一样可回滚可追溯）
- **红线清单**：明确"我们绝不做"的场景（比做更多更重要）

## 六、加分但不强求（更像不同赛道的选择题）

- **向量数据库 / reranker / hybrid search** 原理（做知识问答/RAG必碰）
- **多模态基础**：图像/语音/视频模型的能力边界与成本结构
- **安全合规**：GDPR/个保法思路、模型卡片、数据血缘、审计追踪

这四个点是当前 AI 应用落地中最核心的技术架构 RAG（检索增强生成） 的底层逻辑。

如果把大模型比作一个非常聪明的“实习生”，RAG 就是教会这个实习生“怎么查公司资料库来回答老板的问题”的一套完整工作流。

以下是这四个点的通俗解释：

1. 为什么需要 RAG？（解决幻觉 + 私有知识 + 时效性）

•   大模型的痛点：大模型虽然聪明，但它的知识是“出厂时背下来的”（静态的）。

    ◦   幻觉：它会记混，一本正经地胡说八道。

    ◦   私有知识：它不知道你们公司内部的机密制度、最新的草稿文档。

    ◦   时效性：它不知道今天发生的事，或者你们公司上个月刚改的新流程。

•   RAG 的作用：不给它灌输所有知识，而是教它在回答前先去“查资料”。

    ◦   就像一个学生参加“开卷考试”，模型不需要什么都记得，只要能根据题目迅速在你们的知识库里找到正确的“课本页码”，然后照着念就行了。这样回答既准确，又能随时更新公司内部的最新信息。

2. Embedding / 向量库的作用（相似召回）

这是 RAG 实现“查资料”的技术底座。
•   Embedding（向量化）：把文字翻译成数学坐标。传统的搜索是按“关键词”找（比如搜“苹果”可能搜出水果，也可能搜出手机）。Embedding 能把意思相近的词映射到同一个“坐标区域”（比如“特斯拉股价下跌”和“TSLA 股票走低”在数学空间里离得很近）。

•   向量库：专门存放这些坐标的“高级图书馆”。当用户提问时，系统把问题也转成坐标，去这个图书馆里找距离最近的几本书（Top-K 召回）。这叫“语义搜索”，能真正理解用户的意图。

3. Chunk 切分策略如何影响效果？

•   为什么要切分（Chunking）：大模型一次能看进去的字数是有限的（上下文窗口限制）。你不可能把一整本几百页的《员工手册》全塞给它，那样它不仅看不完，还会被无关信息干扰。

•   怎么切分（策略）：必须把长文档切成一小段一小段的“知识碎片”（Chunks）。

    ◦   切大了：一段里包含太多废话，模型容易被带偏。

    ◦   切小了：一段里缺胳膊少腿，上下文不完整，模型看不懂。

    ◦   产品经理的活儿：决定按多少字切、按段落切还是按标题切，这直接决定了检索出来的资料是不是“刚好够用”。

4. “检索不准 → 生成瞎编”这条链怎么在产品侧缓解？

这句话的意思是：如果第一步“查资料”查错了（比如查到了过期的旧规章），大模型就会拿着错误的资料一本正经地瞎编。
作为产品经理，你不能干等着算法变强，必须在产品功能上给用户兜底。图片里列的四个词就是产品侧的“防瞎编”手段：

•   过滤（Filtering）：在查资料时，利用文档的“元数据”（如生效日期、部门权限）把明显过期的、没权限看的资料直接剔除。

•   重排（Reranking）：系统先粗略找出 10 段相关的资料，再用一个更精细的模型对这 10 段进行“二次打分”，把最靠谱的 3 段挑出来喂给大模型。

•   引用 / 溯源（Citation）：强制要求模型在回答时，必须标明信息是从哪段资料里摘出来的。这样用户一眼就能看出它有没有胡说，也增加了模型瞎编的心理压力。

•   拒答（Refusal）：如果系统查了一圈，发现根本没有靠谱的资料能支撑这个问题（或者查到的资料互相矛盾），产品必须设计成“拒绝回答”，而不是让模型硬着头皮瞎编一个。

好，这次我们把 RAG 从"全景骨架"到"每个关节怎么动"铺开来讲，顺便给你一份真正能顺着学下去的资料清单。你可以把它理解为：RAG 内部地图 + 学习导航仪。

一、RAG 全景骨架：它本质上只有两件事

离线建库（把知识变成可检索的结构）

在线问答（用户提问 → 找相关知识 → 拼上下文 → 让 LLM 基于证据回答）

你可以把 RAG 想象一个 "开卷考试的应试系统"：

阶段 做了什么 类比

离线索引（Indexing） 把你的文档切碎 → 转成向量 → 存进向量库 考前把课本每一段编号、做成"按意思排列的索引卡"

在线检索+生成（Retrieval+Generation） 用户问一个问题 → 去索引卡里找最相关的几段 → 把这几段 + 问题一起交给 LLM 考场上拿到题目 → 翻索引卡找到相关段落 → 照着段落组织答案

这就是为什么 RAG 能解决幻觉/私有知识/时效性——因为答案被"锚定在真实文档片段上"，而不是全靠模型脑补。

1）离线阶段：知识怎么变成"可检索的东西"（最关键却最常被低估）

流程其实是：

原始文档(PDF/Word/网页/Markdown…)
  → ① 清洗 & 解析
  → ② 分块(Chunks)
  → ③ 每块文本 → Embedding 向量
  → ④ 存入向量数据库(Vector DB)，带着原始文本 + 元数据(metadata)


① 清洗 & 解析：这一步决定"垃圾进 → 垃圾召回"

常见坑：
• 直接从 PDF 抽出来的文本里混着 页眉页脚/目录/乱码/表格破碎

• 你不清理，模型就会把页眉当成正文去回答

PM 应该盯的点：系统有没有做"数据源质量门禁"？解析器选的是 PyPDF / PyMuPDF / 商用解析器？表格/图片要不要单独处理？

② 分块（Chunking）：RAG 里最值得你花时间的旋钮

这就是上次说的 Chunk 切分。更完整地说，有这么几种切法：

切法 思路 什么时候用

固定长度切（按 token/字符数硬切） 简单粗暴 快速原型

递归切（优先按 \n\n 再按 \n，最后才硬切） 尽量保全段落语义 最常用默认选项

语义切（按句子边界 + 滑动窗口 + 相似度跳变） 更像"自然边界" 高质量知识库

结构感知切（按 Markdown 标题层级 / 代码函数 / 法条条款） 保留结构信息 技术文档/制度文档/SOP

还要理解两个孪生参数：
• chunk_size：每块多大（常见 256～800 tokens）

• chunk_overlap：块与块重叠多少（让上下文不断档）

⚠️ chunk 太小 → 一块里只剩半句话，LLM 看不懂  

⚠️ chunk 太大 → 召回时把一堆无关废话塞进上下文，模型被噪音干扰或直接挤爆窗口

③ Embedding：把文字变成"几何坐标"

Embedding 模型做的事情：
• 让意思相近的句子在向量空间里"离得近"

• 让意思不同的句子"离得远"

中文场景里最常见的开源选择之一是 BGE（BAAI/bge）系列；API 派常用 OpenAI 的 text-embedding-3-small（便宜、稳）。

④ 向量库（Vector DB）：不是"数据库"，是"近似最近邻检索引擎"

你不用手写算法，但它要能干这件事：

给定用户问题的向量，在几万～几百万个 chunk 向量里，用可接受的时间找出最相似的 Top-K 个

常见选型（按"上手容易→企业级"）：
• Chroma：本地开发首选，轻

• FAISS：Facebook 的向量索引库，快，偏底层

• Qdrant / Weaviate / Milvus：更偏生产级（权限、扩容、持久化、分布式）

2）在线阶段：用户提问后，真正发生的事情（也是翻车的重灾区）


用户 Query
 → [可选] 查询改写/扩展（让问题更好查）
 → Query 也变成 Embedding 向量
 → 去 Vector DB 做相似搜索 → 拿回 Top-K 个候选 chunk
 → [可选] Rerank 重排（精筛）
 → 把 Top 最终几个 chunk 拼成上下文
 → 塞进 Prompt："只根据这些上下文回答，不够就说不知道"
 → LLM 生成答案
 → [可选] 附上引用（把 chunk 的出处原样返回给用户）


为什么"检索不准 → 生成瞎编"是一条致命链？

因为：
1. 如果检索回来的 chunk 本身就不相关 → 你等于给了模型错误教材
2. 如果 Prompt 又写的是"请基于以上资料回答"但没写兜底 → 模型会硬编一个听起来合理但不存在的答案
3. 用户看到答案里有专业术语、有板有眼 → 信了 → 事故

所以产品侧缓解手段，本质上就是在链条的每个环节加保险丝：

环节 保险丝

检索前 查询改写（把口语化标准化）、查询扩展（同义词/子问题）

检索中 混合检索（语义向量 + 关键词 BM25 双通道）、元数据进行时间/来源过滤

检索后 Rerank（把真正相关的顶上去）、上下文压缩（去掉废话）、去重/冲突标注

生成侧 Prompt 强制兜底（"上下文不够就拒绝答"）、要求引用来源

全局 建一个小但高质量的 eval case 集，每次改动都跑一遍看分数掉没掉

二、RAG 的"进阶形态"一张表看清（你就知道大家嘴里的词指什么）

形态 核心差异 什么时候考虑

Naive RAG Query → 一次检索 → 拼上下文 → 生成 原型/MVP

Advanced RAG 检索前改写 + 混合检索 + rerank + 压缩 90%生产系统的甜点区

Modular RAG 模块可插拔：路由、多索引、多路召回、记忆 复杂业务多条知识源

Agentic RAG 模型自己决定是否再查、查什么、能不能答 多跳推理/工具调用类任务

GraphRAG 用知识图谱做实体-关系推理 "张三的主管的部门地址"这类跨文档关联

Multi-Modal RAG 图文表格一起检索 扫描件/报表/带图的制度文档

1.1 必须过关（让你不再被技术细节带着跑）

用产品语言把下面这些讲清楚就行：

• 模型 ≠ 产品：模型输出是概率的、可漂移的；产品要的是可预期的结果 + 可解释失败的路径。

• 训练 / 微调 / 推理（生成）：

拼 Prompt = 你给模型的「系统指令 + 上下文材料 + 用户输入」按固定模板组装成一段完整的话（就像拼一个动态 HTML/组件树）

所以 Schema 就是你和模型之间的"接口协议"，Prompt 就是组装这次请求的"参数+指令"。

一、拼 Prompt：不是一句 prompt 走天下，而是「三层拼装」

真正生产环境里的 Prompt，从来不是一行 "帮我写个文案"，而是像搭积木一样分三层拼出来的：

第①层：System Prompt（系统身份 + 铁律）

定义：这个"角色"是谁、世界观是什么、绝对不能干什么


你是【商品营销文案助手】，专门为运营生成合规的商品描述短文案。
语气：利落、有卖点感、不浮夸。
必须遵守：
- 字数 ≤ 60 字
- 不得出现：价格、折扣承诺（"最便宜""全网最低"）、竞品品牌名
- 不得编造参数（只使用下方【商品信息】给出的字段）
- 如商品信息不足以生成，输出 {"ok": false, "reason": "信息不足"}


第②层：Context（证据材料 / RAG 召回的内容）

定义：模型可以"照着念"的依据——来自知识库、商品库、历史案例、规则文档


【商品信息】
- 标题：轻奢牛皮单肩包 LBS-302
- 类目：箱包 > 女包 > 单肩包
- 材质：头层牛皮
- 主打卖点：轻量（380g）、可放A4、可拆卸肩带
- 品牌调性：简约、通勤、都市

【品牌禁用词库】
最、第一、国家级、顶级……


第③层：User Input（用户真的输入的变量）

定义：每次用户操作带来的动态部分——相当于函数参数


运营选择的风格偏好：干练通勤
运营补充关键词：送闺蜜生日礼物也合适
→ 请生成文案


最终"拼"出来送给模型的完整报文长这样（简化示意）：


[System]
你是商品营销文案助手……（铁律）

[Context]
商品信息：……
品牌禁用词：……

[User]
风格：干练通勤
补充：送闺蜜生日礼物也合适
→ 请生成


这就是 "拼" 的意思：你的产品代码里会有类似拼模板的逻辑——
final_prompt = (
    system_layer()      # 从配置里读，可版本管理
  + context_layer(doc)  # 从RAG/商品库查到的材料
  + user_layer(form)    # 用户填的字段/选的参数
)


⚠️ 对 PM 的关键启发：System Prompt 应该写成"可配置的规范文件"（带版本号），而不是散落在代码注释里的灵机一动。你改一次 System Prompt = 发一次版。

二、输出 Schema：让模型的输出从「散文」变「结构化数据」

为什么必须有 Schema？

如果 Prompt 只写 "请生成文案"，模型可能返回：

🎁 轻奢牛皮单肩包｜380g轻盈随身，A4也能装下，通勤约会都好看～

看着挺美。但问题是：
• 你的前端怎么稳定解析它？

• 你怎么把它存进数据库的某个字段？

• 怎么保证下次不会突然多出两行废话或表情符号炸布局？

• 你怎么让下游的审核/合规程序自动检查？

👉 所以你要强制模型输出 机器可读的结构，通常是 JSON，这就是 Schema。

Schema 怎么写（用上面的例子）

你在 Prompt 里加一句话就够狠：

请严格按照以下 JSON Schema 输出，不要输出任何其他文字、不要加 json``` 标记：

{
  "ok": true,
  "data": {
    "headline": "string, ≤20字, 主标题感",
    "body": "string, ≤60字, 正文文案",
    "tags": ["string", "string", "string"],
    "tone": "干练 温柔
 活力 三选一"
  }
}


模型就会吐出类似：

json
{
  "ok": true,
  "data": {
    "headline": "380g轻量牛皮通勤包",
    "body": "头层牛皮质感在线，A4文件轻松塞入，可拆卸肩带一秒切换手提——上班背得体，下班背不累。",
    "tags": ["通勤", "轻量", "送礼"],
    "tone": "干练"
  }
}


你的前端直接：
js
renderHeadline(json.data.headline)
renderBody(json.data.body)
renderTags(json.data.tags)


这就从"碰运气的自然语言"变成了"可控的产品零件"。

## 三、拼 Prompt + Schema 合在一起，在你的产品链路里长这样


用户点「生成」
  → 前端收集表单字段（风格/长度/用途…）
  → 后端拼 Prompt（System + Context + User变量）
  → 调 LLM，带 output_schema 约束
  → 解析 JSON
  → 校验字段（字数/含禁用词/为空？）
       ├─ 通过 → 展示给用户（带「重新生成」「编辑」「采纳」）
       └─ 不通过 → fallback（给模板备选 / 提示换参数 / 打日志人工抽检）
```

这整张图，就是你 PRD 里应该画的那个「AI生成链路」。

四、落到你的日常工作：你在 PRD 里该怎么写这一节

你可以直接复制这个写法（交互设计师转 PM 最舒服的形式——表格式契约）：

① Prompt 组装说明

层 来源 是否动态 备注

第一部分：核心概念通俗解释（PM视角）

1. 模型 \neq 算法

•   通俗解释：

    ◦   算法是“学习方法”（比如：教小孩认猫，用的是“看颜色”还是“看胡须”的方法）。

    ◦   模型是“学到的本事”（比如：小孩脑子里已经形成了“猫”的概念，能认出猫了）。

•   对工作的启发：

    ◦   产品卖的是模型表现出来的能力（用户体验好不好），而不是你用了什么高大上的算法论文。

    ◦   避坑指南：不要为了用某个新算法而做功能，要为了提升用户体验而引入模型能力。

2. 训练 / 微调 / 推理

这三个词代表了AI产品的不同阶段，成本天差地别：
•   训练 (Training) —— 造发动机

    ◦   状态：贵、慢、需要海量算力。

    ◦   动作：把海量数据喂给机器，让它从0开始学。

    ◦   PM启示：一般大厂做底座才会做训练。绝大多数业务PM不需要碰这个，除非你是做大模型的。

•   微调 (Fine-tuning) —— 给发动机刷漆/改装

    ◦   状态：中等成本，耗时几十分钟到几小时。

    ◦   动作：在已有的大模型基础上，用特定行业的小部分数据（比如几千条法律合同）再学一遍。

    ◦   PM启示：这是ToB产品经理最常碰到的。当你发现通用模型不懂你们公司的黑话时，就需要微调。

•   推理 (Inference) —— 开车

    ◦   状态：消耗的是“油钱”（Token费、服务器费）和“时间”（延迟）。

    ◦   动作：用户输入一句话，模型吐出一句话。

    ◦   PM启示：这是你天天盯着的数据。用户等待多久？花了多少钱？会不会崩？全看推理环节。

3. 参数 / 超参数

•   通俗解释：

    ◦   参数：模型脑子里记住的具体数字（比如“猫”这个词等于 0.8，“狗”等于 0.2）。这是机器自己学出来的，你不用管。

    ◦   超参数：你给机器的“限制条件”和“环境设置”。

•   对工作的启发：

    ◦   把业务目标翻译成超参数。

    ◦   例子：如果你希望回答更保守，就让研发调低“采样温度”；如果你希望回答不啰嗦，就让研发调小“最大生成长度”。

4. 监督 / 无监督 / RL (强化学习)

这是模型学习的三种“姿势”，决定了你需要准备什么数据：
•   监督学习 (Supervised)：手把手教。

    ◦   你需要准备：标准答案（Input: A, Output: B）。

    ◦   场景：最常见的微调方式，比如你给模型看1000个“好评”，它就知道什么是好评了。

•   无监督学习 (Unsupervised)：自己悟。

    ◦   你不需要准备答案，只需要扔给它一堆乱七八糟的文本，它自己找规律。

    ◦   场景：大模型的预训练阶段。

•   RL (Reinforcement Learning)：打游戏拿分。

    ◦   模型做一个动作，你给它打分（+1分或-1分），它为了拿高分不断优化。

    ◦   PM启示：如果你设计了一个“AI打辩论”的功能，用户对回答点赞就是+1分，这就是在用RL思路做产品优化。

5. 泛化 vs 过拟合

这是区分“真智能”和“死读书”的分水岭：
•   过拟合 (Overfitting)：死记硬背。

    ◦   模型把训练数据背下来了。考试遇到原题满分，换个问法就傻眼。你的Demo很惊艳，但上线后效果极差，通常就是过拟合了。

•   泛化 (Generalization)：举一反三。

    ◦   模型掌握了底层的规律，遇到没见过的新问题也能答对。

•   PM启示：判断Demo的水分。如果研发给你的Demo全是训练过的数据，千万别信。一定要用全新的问题测试它。

6. 不确定性

•   通俗解释：AI本质上是一个概率计算器。它每次回答都是在做“猜词游戏”（下一个词是A的概率80%，是B的概率20%）。

•   对工作的启发（最重要的一点）：

    ◦   作为设计师/PM，绝对不能假设AI是对的。

    ◦   你要时刻准备Plan B（兜底方案）：如果它瞎编怎么办？如果它拒答怎么办？如果它太慢怎么办？这就是我们之前说的“容错设计”。

第二部分：JSON 数据到底是什么意思？

作为交互设计师，你可能觉得 JSON 很抽象。我们换个角度，不用技术思维，用“乐高积木说明书”的思维来理解。

1. 为什么要用 JSON？（解决的痛点）

假设你让AI写一篇文章。
•   以前的做法（自然语言）：AI 可能给你一大段文字，有时候开头空两格，有时候不空；有时候分段很清楚，有时候挤成一坨。前端开发很难排版，因为你给的格式是不确定的。

2. JSON 的形象比喻：标准化的快递包裹

JSON 就是一种严格的包裹打包格式。不管里面装的是什么，外面的包装盒必须是方的，接口必须是标准的。

举个例子：
你让 AI 生成一个“用户信息”。

•   ❌ 不好的输出（自然语言，乱七八糟）：

    > “他的名字叫张三，年龄大概25岁吧，喜欢打篮球和游泳，住在北京，邮箱是 zhangsan@qq.com 不对是 gmail 吧...”

•   ✅ 好的输出（JSON格式，井井有条）：
    {
      "name": "张三",
      "age": 25,
      "hobbies": ["篮球", "游泳"],
      "city": "北京",
      "email": "zhangsan@gmail.com"
    }
    

3. 怎么看懂这个结构？（交互设计师专属解读）

你不需要会写代码，你只需要把 JSON 看成“有明确格子的表单”：

•   大括号 { } = 一张表。

•   双引号里的字（如 "name"） = 格子的标题（变量名）。

•   冒号后面的内容 = 格子里的具体内容。

•   方括号 [ ] = 这是一个列表/数组（比如多选框的结果）。

对应到你的工作流程中：

1.  以前（交互设计）：你画原型，前端问你：“输入框填什么？” 你说：“填个名字。” -> 前端写死一个框。
2.  现在（AI产品经理）：你拼 Prompt，后端问你：“模型输出什么？” 你说：“输出一个 JSON，里面必须包含一个叫 name 的字符串变量。” -> 后端直接把这个变量塞到页面的任何地方（标题、卡片、图表里）。

作为转岗的 AI PM，你现在需要建立这样的心智模型：
用户提问 -> 模型推理 -> 输出 JSON (结构化数据) -> 前端渲染 (交互展示)。

1）Schema 怎么读（发音）

• 英式/美式常见读法：/ˈskiːmə/ → 谐音 “skee-muh（斯ki-嘛）”  

• 日常口头也常直接说：“输出结构 / 输出格式 / JSON 的字段契约”（开会时这么说大家都懂）

2）Schema 到底是什么（用你的交互语言翻译）

你以前做 UI 时一定会定：字段名、必填/选填、长度、类型、枚举值（下拉选项）  
比如一个“商品卡片”要显示：

• 标题（文字，最长 20 字）

• 价格（数字，单位元）

• 标签（最多 3 个，只能从固定集合里选：新品/热销/限量）

Schema 就是：把这种“字段规则”写给模型看 + 写给系统用。

也就是说：你不是让模型“自由发挥写一段话”，而是命令它“填一张固定表格”。

3）为什么有了 Prompt 还要 Schema（这是关键）

Prompt 负责：告诉模型 做什么 / 按什么规矩做

“你是文案助手；不得编造参数；字数限制；禁用词…”

Schema 负责：规定 输出必须长什么样

“结果必须是一个 JSON，里面一定有 headline、body、tags，类型是字符串/数组，不能多不能少。”

打个比方：
• Prompt = 你给员工的任务说明 + 制度手册  

• Schema = 你规定交差时必须交 Word 不行，必须交 Excel，而且列名必须是 A 列 title、B 列 desc

没有 Schema 会发生什么？
• 有时输出带 emoji、有时多两句废话、有时包在 `json` 里、有时直接写“您好，已为您生成…”

• 你的前端只能“猜”怎么解析 → 一上线就裂开（要么空白、要么样式炸掉）

4）Schema 最常用形态：JSON Schema（最小白话版）

下面这个就是最常见的 Output Schema：模型只能按这个壳子吐数据。
{
  "headline": "string（≤20字）",
  "body": "string（≤60字）",
  "tags": ["string"],
  "tone": "干练 | 温柔 | 活力"
}


你把它“写进 Prompt 里”时，通常会说得更硬一点（伪代码版）：

你必须只输出一个 JSON 对象，字段如下：
- headline: string, 主标题, ≤20字
- body: string, 正文文案, ≤60字
- tags: string[], 最多3个, 只能从 [通勤,送礼,轻奢,极简,复古] 里选
- tone: enum, 只能是 干练/温柔/活力

不要输出 json``` 标记，不要输出解释文字。


模型就会吐出类似：

json
{
  "headline": "380g轻量牛皮通勤包",
  "body": "头层牛皮质感在线，A4文件轻松塞入，可拆卸肩带一秒切换手提——上班背得体，下班背不累。",
  "tags": ["通勤", "轻奢"],
  "tone": "干练"
}
```

前端/后端就可以稳稳地：
• card.title = data.headline

• card.desc = data.body

• 把 tags 画成标签胶囊

5）Schema 对你当 AI PM 的真实意义：它是“接口契约”（你要盯的三件事）

① 你可验收（不靠感觉）

验收条件不再是“看起来不错”，而是：
• 是不是合法 JSON？

• 必填字段在不在？

• 长度/枚举有没有越界？

• 有没有把禁止内容塞进来？

② 你能兜底（体验不崩）

当解析失败或校验不过时，你规定走什么分支：
• 重试一次

• 或 fallback 到模板文案

• 或提示用户“信息不足，请补充：材质/使用场景…”

③ 你能迭代（改 prompt 不再心虚）

因为你约定了输出形状，每次改 Prompt 你都能跑回归 case：  
“这次改完，headline 还在吗？越界率升了吗？解析失败率升了吗？”

传统交互设计你最熟的就是画异常流。AI产品里异常流是主战场。

你在PRD里加一节，别的PM不写、但你写，就成了你的签名：

§ 异常与降级策略（AI专用）

> 异常 条件 用户看到什么 系统做什么

--- --- --- ---

检索空 召回<阈值或top分<0.6 「没找到足够资料，请补充XX」 不打模型，直接return

生成超时 >8s无首token skeleton→「稍后再试/用精简版」 切fallback模板

解析失败 JSON无效/缺必填字段 「重新生成」按钮亮起 自动retry一次

安全拦截 命中敏感/越权 统一提示，不暴露规则细节 log+上报

低质输出 含禁用词/自相矛盾 拦截+原因code 进人工抽检队列

这一节放进去，研发会松口气——因为有人终于把边界划清了。

抓手4：【Eval最小闭环】—— 你从"意见层"沉到"证据层"

这是UX→AI PM最关键的跃迁：从"我觉得效果不行"变成"我的case集说它不行"。

你不需要会训练模型，你只需要做这一件事👇

Week 1–2 动作：建一个 30～50 条的 eval sheet

ID 输入（用户会怎么问/填） 期望行为 红线（绝不能发生） 上次通过？

Q01 "XX产品的保修期多长" 从文档召回+引用 不能编造数字 ✅/❌

Q02 只给标题没给参数→生成文案 拒答/要补充 不能瞎编参数 ✅/❌

R01 输入含竞品名 不出现在输出 竞品名不提/脱敏 ✅/❌

然后跟研发说一句话：

"我每次改prompt或换chunk策略，跑这50条当回归。不通过的改动不上线。"

这句话一出口，你的角色就不可逆地从设计师变成PM了。

抓手5：【语言转换】—— 学会用PM/业务的货币说话

你的UX洞察必须翻译成两种货币，才能在会上流通：

你的UX直觉 翻译成PM/业务货币

"用户看到一长串无来源文字会慌" "无引用→采纳率掉→人工复核成本↑→我们需要引用标签/溯源"

"加载超过5秒不给反馈，用户以为坏了" "p95首token>4s→流失陡增→需要streaming+SKELETON，SLA设Xs"

好，这句确实是整条链路里最「工程感」的一句，所以你卡住很正常。我们把整句话拆成四个词，用你已有的 Prompt 心智模型（「给模型塞材料让它照着念」）来重新翻译一遍——不讲虚的。

先把大前提钉死：RAG 的检索 = 帮模型「找参考资料」

你之前理解了 Prompt = 组装指令 + 材料 + 用户问题。

其中有一类材料不是你手写的，而是从你们自己的资料库里临时找出来的——比如：
• 公司制度文档

• 产品手册

• 客服知识库

• 合同条款碎块

检索（RAG）做的就是一件事：

用户问完问题之后，先去资料库里搜一圈，把「最可能回答这个问题的那几段原文」捞出来，再塞进 Prompt 给模型。

所以它不是一个黑盒魔法，本质就是一个「智能版搜索引擎 → 取前几名 → 把结果当材料用」的过程。

逐词拆：「召回 Top-K 块 → 没召回着？ → 走拒答/澄清」

① 召回（Retrieve / 召回）

意思是：从资料库里「召」出来一批可能相关的资料片。

类比你去图书馆找书：
• 用户问：「产假怎么请？」

• 系统就去资料库里翻：哪些段落跟「产假 / 请假流程 / HR制度」最相关？

这一步叫召回——宁可多召一些候选，后面再筛。

② Top-K 块（Top-K chunks）

刚才说过资料会被切成一块一块的「知识碎片」（chunk）。

Top-K = 我只取「最相关的前 K 块」，比如 Top-3 或 Top-5。

• Top-1：只拿最相关的一块（容易漏上下文）

• Top-3/5：常见起步配置（够用但别太多）

• Top-10：太多 → 塞进 Prompt 会变杂、模型被噪音带歪

所以这句话：

召回 Top-K 块

翻译成人话就是：

去资料库里找，把跟用户问题最相关的 K 段原文先捞出来，准备当「参考资料」塞给模型。

③ 没召回着？（关键：这不是模型的问题，是「资料/匹配」的问题）

所谓「没召回着」，就是下面两种情况之一（或叠加）：

1. 资料库里压根没有相关内容
   • 用户问：「咱们公司明年去哪团建？」

   • 资料库只有制度/产品手册 → 匹配不到 → 召回得分全部很低

2. 有内容，但没匹配上（最常见、最气人）
   • 用户问法跟文档写法不一样  

     ◦ 文档写：「女性职工生育假申请流程」

     ◦ 用户问：「怀孕了怎么请假？」

   • 或者 chunk 切得太碎，关键信息被切断了

   • 或者检索只靠关键词，没理解语义

结果都一样：捞出来的 Top-K 块质量很差 / 数量不够 / 相关度分数太低——等于没找到靠谱材料。

④ 走拒答 / 澄清（产品兜底分支）

这里就是 AI PM 的真正战场：你不能在「没找到资料」时还让模型硬编答案。

所以你要做一个分支判断，伪代码写成文字是这样的：

如果（召回的块 相关度 < 阈值）或者（召回数量 == 0）：
    走 拒答 / 澄清
否则：
    把块塞进 Prompt → 生成


「拒答」是什么？

直接告诉用户：我没法回答，但说得专业且不冷：

• ❌ 别让模型说：「作为一个AI语言模型……我不知道」

• ✅ 产品侧写死统一话术：

  • 「当前知识库暂未收录该问题相关信息，可尝试换个表述，或联系 XX」

  • 「我找到的资料不足以确认答案，为避免误导，暂不作答」

「澄清」是什么？

不是直接拒，而是反问用户一句，把检索救回来：

• 「您说的是『产假』还是『育儿假』？请确认后再问」

• 「没找到『XX产品』的说明，您是指『全称 XXX』吗？」

• 给出 2～3 个候选（本质是拿召回命中的模糊结果做选择题）

用一个你一秒能代入的完整例子（吃透它就通了）

场景：内部知识库问答（HR制度助手）

用户问：

「产假怎么请？」

背后发生的事：

1. 检索（RAG）
   • 系统去向量库搜：跟「产假 / 请假 / 生育」最相关的段落

2. 召回 Top-3 块
   • 块A：《女性职工生育相关假期申请流程…》

   • 块B：《所需材料：诊断证明…》

   • 块C：得分太低/不相关 → 被丢掉或排在后面

3. 质量门（关键）
   • 块A 的「相关度分数」0.82（高），块B 0.71 → 过阈值 → 可以用

4. 拼 Prompt

   [System] 你是HR制度助手，只能根据上述资料回答…
   [Context]
   块A原文：……
   块B原文：……
   [User] 产假怎么请？
   
5. 生成 → 输出带引用的答案

但如果用户问：

「明年 Q2 的团建地点定哪了？」

检索一遍 → 最高分只有 0.31（远低于阈值） → Top-K 其实「没召回着靠谱材料」

此时你不送模型（或者送了也会编）→ 走拒答/澄清：

• 拒答版：

  > 「该问题超出当前制度知识库范围，可咨询行政部 XXX」
• 澄清版：

  > 「没匹配到『团建』相关制度信息，您是想问『团队活动经费申请流程』吗？」

你作为 UX-trained AI PM，从这学到的不是技术，是两条铁律

铁律1：召回质量 × 引用透明 = 可信度下限

没召回着还硬生成 = 幻觉事故的温床。  
所以你在 PRD 里要写死：

• 相关度阈值（比如 0.6）

• 最少召回块数（比如至少 1 块 ≥ 阈值）

• 不满足就进拒答/澄清分支

铁律2：拒答不是失败，是保护体验

用户更恨的是「看起来很权威的错答案」，不是「老实说我没找到」。  
所以拒答文案、澄清话术、下一步按钮（联系人工 / 换问法 / 打开表单）是你做 UX 的地方。
