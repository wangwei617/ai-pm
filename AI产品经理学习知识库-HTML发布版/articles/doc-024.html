Skill 能力单元与暴露策略

Skill 是一个动词的实现体，Agent 是决定什么时候调用这些动词的调度者。

你会看懂

能用一句话解释「Skill 能力单元与暴露策略」为什么值得学。

0. 一句话抓住本质

Skill 是一个动词的实现体，Agent 是决定什么时候调用这些动词的调度者。

1. 一句话定义

Skill 是系统级的可调用能力单元。它做一件明确的事，有清晰输入、输出、权限和风险等级。

Agent 是调度者，决定先调用哪个 Skill、后调用哪个 Skill。

用户直接使用 Skill，本质上是产品把某个安全、单一、可校验的能力包装成按钮、指令或表单入口。

2. Skill 更像汽车里的什么

如果 LLM 是发动机，Agent 是驾驶员加行车电脑，Skill 更像车载系统里的附件功能：

  • 雨刮器。
  • 空调。
  • 电动窗。
  • 充气泵。
  • 绞盘。
  • 外放电插座。

它们的共同点是：

  • 做一件事，做完就停。
  • 有明确输入和输出。
  • 可以被用户按钮触发。
  • 也可以被自动驾驶/辅助驾驶逻辑触发。
  • 有些可以直接暴露，有些必须锁在系统内部。

所以：

Agent : Skill = 调度者 : 能力单元

3. Skill 的架构视角

工程上，Skill 通常会被设计成 Tool / Function。

示例：

{
  "name": "query_order",
  "description": "查询订单状态，只读",
  "parameters": {
    "order_id": "string"
  },
  "risk": "low",
  "allow_user_direct": true
}

这意味着：

  • Agent 可以调用它。
  • Router 可以在意图确定时直通调用它。
  • UI 可以把它包装成按钮或表单。
  • Harness 可以根据风险决定是否允许直通。

Skill 最好一开始就按工具规格设计，否则未来 Agent 很难接住。

4. Agent 调用 vs 用户直通

Agent 路径

用户目标
→ LLM 规划
→ 选择 Skill
→ 带参数调用
→ 获取结果
→ 继续规划或最终回复

适合：

  • 多步骤任务。
  • 需要条件判断。
  • 需要多个 Skill 串联。
  • 需要工具结果影响下一步。

用户直通路径

用户点击按钮/输入指令/提交表单
→ Router 确认意图
→ 参数校验
→ 调用 Skill
→ 渲染结果

适合：

  • 单一动作。
  • 意图确定。
  • 低风险。
  • 参数可校验。
  • 无副作用或可撤销。

用户直通不是让用户直接碰工具底层，而是产品给了一个受控入口。

5. 用户直接用 Skill 的产品形态

产品形态本质示例
斜杠命令路由直通 Skill/订单状态 → 查询订单
按钮路由直通 Skill「查物流」按钮
表单提交参数来自表单填订单号后查询
预览卡确认Agent 预览后调用 Skill改地址前点确认

共同点：

  • 入口是产品给的。
  • 参数经过清洗和校验。
  • Skill 在白名单里。
  • 风险等级已判断。
  • 必要时有确认和审计。

6. Skill 暴露策略

Skill 类型汽车等价是否允许用户直通暴露方式控制措施
只读查询雨刮器/空调读数允许按钮、指令、表单权限校验、日志
生成类电动窗允许生成卡片、编辑、采纳Schema、禁用词、反馈
通知/消息喇叭/转向灯谨慎允许模板化预览模板、预览、确认
写数据绞盘/差速锁不允许直通Agent 预览后确认二次确认、审计、可撤回
内部工具高压维修接头绝不暴露内部 Harness 使用内部权限、审计

一句话：

能直通的 Skill 必须低风险、可校验、无副作用或可撤销。
有副作用的 Skill 必须藏在 Agent/Workflow 后面，并装刹车。

6.1 真实业务案例：订单平台的 Skill

假设订单平台有 5 个 Skill：

Skill类型是否可直通原因
query_order只读查询可以低风险，参数可校验
generate_reply生成类可以输出可编辑和采纳
send_message消息谨慎必须模板化和确认
change_address写数据不可以有副作用，必须预览和审计
read_system_log内部工具不可以高权限内部能力

用户点击「查物流」，可以直通 query_order。

但用户说「帮我改地址并通知客户」，必须进入 Agent/Workflow：

查订单 → 生成改地址预览 → 用户确认 → 修改地址 → 生成短信预览 → 用户确认 → 发送

同样是 Skill，暴露方式完全不同。

6.2 反例：把所有 Skill 都给用户直通

错误做法：

把 query_order、send_message、change_address、read_system_log 都做成按钮或指令。

后果：

  • 用户误点导致数据被修改。
  • 高权限工具被普通用户看到。
  • 缺少预览和确认，消息误发。
  • 没有审计，事故难以追责。

Skill 暴露策略不是 UI 选择，而是风险治理。

7. PM 要定义的字段

每个 Skill 至少要定义：

字段说明
nameSkill 名称
description做什么
input_schema输入字段和约束
output_schema输出字段和格式
risk_level风险等级
side_effect是否有副作用
reversible是否可撤回
permission_required权限要求
allow_user_direct是否允许用户直通
require_confirmation是否需要确认
audit_required是否需要审计
owner负责人

8. Skill、Tool、Agent、Workflow 的区别

概念解释
Skill做一件事的能力单元
Tool / FunctionSkill 在工程里的调用形态
Agent决定调用哪些 Skill，以及调用顺序的调度者
Workflow预设流程，按固定或半固定步骤编排 Skill
Harness控制 Skill/Agent 能做什么、如何确认、如何审计的外壳

9. 常见误区

  • 以为 Skill 只能给 Agent 用。
  • 以为用户直通 Skill 就是把工具列表暴露给用户。
  • 所有 Skill 都做成自由聊天入口。
  • 写操作 Skill 允许直通。
  • 没有给 Skill 标风险等级。
  • 没有定义输入输出 Schema。
  • 没有权限、确认和审计。

9.1 PRD 可粘贴写法

所有 Skill 必须登记风险等级、输入输出 Schema、副作用、是否可撤回、是否允许用户直通。
低风险只读 Skill 可以通过按钮、指令或表单直通。
有副作用 Skill 不允许直通，必须通过 Agent/Workflow 执行，并要求预览、确认和审计。
内部工具不对普通用户外显。

Skill 暴露策略表：

Skill风险副作用允许直通入口控制措施

10. 人话总结

Skill 是动词的实现体。

Agent 是决定先干哪件、后干哪件的调度者。

用户直接用 Skill，是把某个安全、单一、可校验的动词包装成按钮、指令或表单给用户按。

但按钮背后仍然是受控系统调用，不是把底层工具交给用户随便用。

11. 一句可复述总结

Skill 可以被 Agent 调，也可以被用户通过按钮直通；但能不能直通，不看技术能不能做，而看风险、权限、副作用和可撤回性。

实战化补充

把「Skill 能力单元与暴露策略」用到真实工作里

这一段不是补术语，而是帮你把文章从“读懂”推进到“能用”。公开分享时，读者最需要的通常不是更多概念，而是知道下一步该怎么做。

轻松理解

Agent 不是更会聊天的人，而是一个在受控循环里做事的调度员：想一步、调工具、看结果、再决定下一步。

业务小案例

客服改地址场景里，Agent 不能直接“想改就改”。它应该先查订单、判断状态、生成改动预览、让用户确认，再调用改地址工具并留下 Trace。

常见翻车点

  • 把高风险工具直接暴露给模型
  • 只设计 happy path，没有超时、失败和人工接管
  • 把路由、权限和审计都写进 Prompt

建议沉淀的交付物

  • 工具白名单
  • Skill 登记表
  • Agent Loop 流程图
  • Gateway 路由规则
  • 审计与人工确认策略

读完马上做

选一个你熟悉的业务动作，拆成 5 个 Skill，并给每个 Skill 标注风险等级和是否需要人工确认。

一句可复述总结

学习「Skill 能力单元与暴露策略」的重点，不是背定义，而是把它变成可判断、可设计、可验收、可复盘的产品工作。
