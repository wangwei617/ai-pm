这份模板用于盘点平台里的 Skill 节点，并判断哪些可以用户直通，哪些必须经过 Agent/Workflow，哪些必须内部锁死。

1. Skill 清单

只读/生成/消息/写数据/内部工具有/无是/否

2. 暴露策略

Skill 名称允许 Agent 调用允许用户直通推荐入口是否需确认是否需审计原因
是/否是/否按钮/指令/表单/预览卡/不外显是/否是/否

3. 分类规则

可直通暴露

满足：

  • 低风险。
  • 意图确定。
  • 参数可校验。
  • 无副作用或可撤回。
  • 用户能理解结果。

适合入口：

  • 按钮。
  • 斜杠命令。
  • 表单。

必须经过 Agent/Workflow

满足任一：

  • 需要多步骤判断。
  • 需要多个 Skill 串联。
  • 有副作用。
  • 需要预览和确认。
  • 需要根据工具返回继续决策。

适合入口：

  • Agent 对话。
  • 工作流。
  • 预览卡确认。

必须锁死不外显

满足任一：

  • 内部调试工具。
  • 系统日志查询。
  • 安全扫描。
  • 高权限操作。
  • 用户无法理解或无法判断风险。

适合入口：

  • 内部 Harness。
  • 后台管理。
  • 不对普通用户展示。

6. 上线验收

  • [ ] 每个 Skill 都有 owner。
  • [ ] 每个 Skill 都有输入输出 Schema。
  • [ ] 每个 Skill 都标注风险等级。
  • [ ] 每个 Skill 都标注是否有副作用。
  • [ ] 每个 Skill 都明确是否允许用户直通。
  • [ ] 写数据 Skill 不允许用户直通。
  • [ ] 内部工具不对普通用户外显。
  • [ ] 高风险 Skill 有确认和审计。
  • [ ] 所有 Skill 均纳入白名单管理。
实战化补充

把「Skill 盘点与暴露策略模板」用到真实工作里

这一段不是补术语，而是帮你把文章从“读懂”推进到“能用”。公开分享时，读者最需要的通常不是更多概念，而是知道下一步该怎么做。

轻松理解

Agent 不是更会聊天的人，而是一个在受控循环里做事的调度员：想一步、调工具、看结果、再决定下一步。

业务小案例

客服改地址场景里，Agent 不能直接“想改就改”。它应该先查订单、判断状态、生成改动预览、让用户确认，再调用改地址工具并留下 Trace。

常见翻车点

  • 把高风险工具直接暴露给模型
  • 只设计 happy path，没有超时、失败和人工接管
  • 把路由、权限和审计都写进 Prompt

建议沉淀的交付物

  • 可直接复制的模板结构
  • 字段使用说明
  • 评审时的检查问题
  • 交付后的维护责任人

读完马上做

选一个你熟悉的业务动作，拆成 5 个 Skill，并给每个 Skill 标注风险等级和是否需要人工确认。

一句可复述总结

学习「Skill 盘点与暴露策略模板」的重点，不是背定义，而是把它变成可判断、可设计、可验收、可复盘的产品工作。
