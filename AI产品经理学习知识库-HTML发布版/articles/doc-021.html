RAG 进阶与产品验收

RAG 不是「给模型接个知识库」这么简单，而是让模型在回答前先拿到可信证据，并让用户能追溯这些证据。

0. 一句话抓住本质

RAG 不是「给模型接个知识库」这么简单，而是让模型在回答前先拿到可信证据，并让用户能追溯这些证据。

RAG 的产品质量下限，取决于两件事：

召回质量 × 引用透明

1. RAG 的两阶段

RAG 本质上分成两件事：

  • 离线建库：把知识变成可检索的结构。
  • 在线问答：用户提问后检索相关知识，再让模型基于证据回答。

可以把 RAG 理解为开卷考试系统。离线阶段是把教材整理成索引卡，在线阶段是根据题目翻索引卡，再组织答案。

如果资料找错了，模型就像拿错课本考试。它可能答得很流畅，但答案仍然是错的。

2. 离线建库

文档清洗与解析

原始文档可能来自 PDF、Word、网页、Markdown、表格或扫描件。

常见问题：

  • 页眉页脚混入正文。
  • 表格破碎。
  • 目录重复。
  • 图片内容丢失。
  • PDF 抽取乱码。
  • 文档版本过期。

PM 要关注是否有数据源质量门禁，而不是把所有文档直接丢进知识库。

Chunk 切分

Chunk 是知识库中的最小检索片段。

常见切分方式：

切法思路适用场景
固定长度切按字符或 token 硬切快速原型
递归切优先按段落、换行、句子切通用默认
语义切按语义边界和相似度变化切高质量知识库
结构感知切按标题、条款、函数、章节切技术文档、制度、SOP

两个关键参数：

  • chunk_size：每块多大。
  • chunk_overlap：相邻块重叠多少。

切得太小，上下文不完整。切得太大，噪音太多，还会挤占上下文窗口。

Embedding

Embedding 把文本转换成向量，让系统可以按语义相似度检索。

它解决的是「意思相近」的问题，而不是简单关键词匹配。

向量库

向量库用于在大量 chunk 中找出和用户问题最相似的内容。

常见选型包括 Chroma、FAISS、Qdrant、Weaviate、Milvus 等。

PM 不一定要选具体技术，但要问清：

  • 数据规模多大。
  • 是否需要权限。
  • 是否需要持久化。
  • 是否需要多租户。
  • 是否需要生产级稳定性。

3. 在线问答

典型流程：

用户 Query
→ 查询改写或扩展
→ Query 向量化
→ 向量库相似搜索
→ 返回 Top-K 候选 chunk
→ Rerank 重排
→ 上下文压缩和去重
→ 拼进 Prompt
→ 模型基于上下文回答
→ 展示答案和引用来源

4. 为什么检索不准会导致生成瞎编

如果检索回来的资料不相关，模型就像拿错教材答题。

风险链路是：

检索不准 → 上下文错误 → 模型基于错误材料生成 → 用户信任专业答案 → 业务事故

产品侧必须在每个环节加保险：

  • 检索前：查询改写、查询扩展、意图识别。
  • 检索中：混合检索、元数据过滤、权限过滤、时间过滤。
  • 检索后：Rerank、去重、冲突标注、上下文压缩。
  • 生成侧：强制引用、证据不足则拒答。
  • 全局：评测集和错误案例库。

4.1 真实业务案例：HR 制度助手

用户问：

产假怎么请？

系统应该：

  1. 将问题路由到 RAG 链路。
  2. 从 HR 制度知识库召回相关 chunk。
  3. 检查 Top-K 召回质量。
  4. 如果召回到《生育假申请流程》和《请假材料要求》，再让模型回答。
  5. 答案附引用，用户能展开原文。

如果用户问：

明年 Q2 团建地点定哪了？

而知识库里没有相关资料，系统应该拒答或澄清，而不是让模型编一个听起来合理的地点。

好的拒答不是失败，而是避免用户被错误答案误导。

5. Top-K 召回质量门

Top-K 是系统从知识库中召回的前 K 个相关 chunk。

产品上要把「召回是否足够可靠」写成明确规则，而不是让模型自己判断。

判定规则示例

判定项示例规则处理方式
召回数量召回数量 = 0不生成，进入拒答
最高相关度top1 score < 0.6不生成，进入拒答或澄清
有效 chunk 数score >= 0.6 的 chunk 少于 1 个不生成，进入拒答或澄清
资料冲突高相关 chunk 结论互相矛盾标注冲突，要求用户选择或转人工
权限过滤后为空用户无权限查看召回资料不展示资料，提示权限不足

阈值不是固定真理，要通过评测集和线上日志调整。关键是 PRD 必须明确：什么情况不允许模型硬答。

拒答和澄清的区别

拒答是系统确认没有足够材料支撑回答。

适合：

  • 知识库没有相关内容。
  • 相关度明显低。
  • 用户问题超出系统范围。
  • 涉及红线或权限限制。

澄清是系统发现用户问题可能可以回答，但需要补充或确认意图。

适合：

  • 用户表述模糊。
  • 有多个相近候选。
  • 产品名、政策名、时间范围不明确。
  • 检索到低置信但相近的资料。

UX 文案示例

拒答文案：

我找到的资料不足以确认答案。为避免误导，暂不作答。
当前知识库暂未收录该问题相关信息，可尝试换个表述，或联系对应负责人。

澄清文案：

你说的是「产假」还是「育儿假」？请选择后我再继续查找。
没找到「XX 产品」的说明。你是指「XXX 产品全称」吗？

下一步动作

拒答或澄清不应该是死路。可以提供：

  • 换个问法。
  • 补充信息。
  • 联系人工。
  • 打开相关表单。
  • 查看可能相关资料。
  • 提交知识缺口反馈。

两条铁律

  • 召回质量乘以引用透明，是 RAG 可信度的下限。
  • 拒答不是失败，是保护用户体验。

5.1 反例：没有质量门会怎样翻车

错误做法：

无论召回结果相关不相关，都塞给模型生成答案。

后果：

  • 模型拿无关资料回答。
  • 用户看到有引用，以为更可信。
  • 旧制度、错制度或无权限制度被包装成正式答案。
  • 业务方以为是模型幻觉，其实根因是检索质量门缺失。

AI Product Owner 要追问的不是「模型为什么答错」，而是：

  • 检索到了什么？
  • 分数是多少？
  • 是否过了质量门？
  • 引用是否真的支撑答案？
  • 没过质量门为什么还生成？

6. RAG 进阶形态

形态核心差异什么时候考虑
Naive RAG一次检索后生成原型和 MVP
Advanced RAG查询改写、混合检索、Rerank、压缩大多数生产系统
Modular RAG多索引、多路召回、路由、记忆多知识源复杂业务
Agentic RAG模型决定是否再查、查什么多跳推理和工具调用
GraphRAG用知识图谱处理实体关系跨文档关系推理
Multi-Modal RAG图文表格一起检索扫描件、报表、图文制度

7. PM 视角验收清单

上线前至少确认：

  • 知识来源是否可信。
  • 文档是否清洗。
  • Chunk 策略是否经过对比。
  • 是否能查看每次检索返回的原文。
  • 是否有引用来源。
  • 找不到答案是否拒答。
  • 过期文档是否过滤。
  • 用户权限是否影响检索范围。
  • 是否有 RAG 评测集。
  • 每次改 chunk、检索、rerank、prompt 是否跑回归。
  • 是否设置 Top-K 召回质量门。
  • 是否有拒答和澄清分支。
  • 拒答是否提供下一步动作。

7.1 PRD 可粘贴写法

当 RAG 召回结果不满足质量门时，系统不得进入答案生成。
若召回数量为 0，进入拒答。
若 top1 score 低于阈值但存在相近候选，进入澄清。
若召回资料冲突，标注冲突并引导人工确认。
所有答案必须展示引用来源。

RAG 质量门：

判定项阈值或规则通过不通过
召回数量>= 1进入生成拒答
top1 score>= 0.6进入生成拒答/澄清
引用支持引用能支撑结论展示答案标记失败
权限过滤用户有权限展示引用权限提示
资料冲突无冲突展示答案冲突提示/转人工

8. 常见质量指标

  • 检索命中率。
  • Top-K 相关率。
  • 召回质量门通过率。
  • 引用准确率。
  • 答案准确率。
  • 拒答正确率。
  • 澄清成功率。
  • 幻觉率。
  • 用户采纳率。

9. 最小实践路线

可以用一个周末跑通：

  1. 准备 3-5 份真实文档。
  2. 跑通 Naive RAG。
  3. 手动测试 10-20 个问题。
  4. 记录每个问题检索回来的 chunk。
  5. 对比两组 chunk_size 和 overlap。
  6. 加入兜底指令：上下文不足就说不知道。
  7. 记录检索错、生成错、引用错的案例。

10. 一句可复述总结

RAG 不是让模型多知道一点，而是让模型回答前先拿证据；证据找不到、找错了、没权限，就不应该硬答。

实战化补充

把「RAG 进阶与产品验收」用到真实工作里

这一段不是补术语，而是帮你把文章从“读懂”推进到“能用”。公开分享时，读者最需要的通常不是更多概念，而是知道下一步该怎么做。

轻松理解

可以把它理解成给模型配一个会翻资料的研究助理：先找证据，再组织回答，证据不够就别硬答。

业务小案例

例如 HR 制度助手回答“年假怎么折算”。真正的产品重点不是让模型说得顺，而是确认它查到了正确制度版本、权限允许展示，并且引用能支撑结论。

常见翻车点

  • 只看答案像不像，不看引用是否支撑结论
  • 把召回失败当成模型不聪明
  • 没有拒答和澄清机制，导致用户越问越不信

建议沉淀的交付物

  • 知识库切分策略
  • Top-K 质量门
  • 引用展示规范
  • 拒答/澄清文案
  • RAG 验收样本集

读完马上做

拿你公司的一份制度或 FAQ，设计 10 个真实问题，并标注每个问题应该引用哪一段材料。

一句可复述总结

学习「RAG 进阶与产品验收」的重点，不是背定义，而是把它变成可判断、可设计、可验收、可复盘的产品工作。
